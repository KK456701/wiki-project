# 项目 Agent 协作约束

本文件用于约束后续所有 Agent、脚本生成器、文档生成器和协作者在本项目中的输出习惯。项目面向医院本地部署和医务/质控人员使用，默认沟通和交付语言必须是中文。

## 语言规范

1. 默认使用中文编写项目文档、README、部署说明、操作手册、测试说明、设计方案、变更记录和用户可见文案。
2. 默认使用中文编写提示词、Agent 输出模板、前端提示语、错误提示、诊断结论和审批说明。
3. 默认使用中文编写代码注释。只有在解释第三方英文术语、协议字段或库行为时，才保留英文原词。
4. 不要生成整段英文说明给医院用户、管理员或实施人员。必要英文术语后面应补中文解释。
5. 如果必须引用英文原文、API 字段、包名、命令、环境变量、数据库字段或协议名称，应保留原样，但周围解释必须使用中文。

## 允许保留英文的场景

- Python/JavaScript 函数名、变量名、类名、模块名。
- API 路径、JSON 字段名、HTTP header、MCP tool 名称。
- SQL 关键字、数据库字段名、表名、索引名。
- 第三方库名称，例如 FastAPI、LangGraph、Ollama、DBHub、SQLAlchemy。
- 配置项名称和环境变量，例如 `runtime_db_url`、`OLLAMA_BASE_URL`。
- 命令行命令，例如 `python -B -m uvicorn app.api.main:app`。

## 文档风格

1. 面向实际落地说明，不写空泛概念。
2. 先说明“怎么用”，再说明“为什么”。
3. 对医院部署、内网环境、只读数据库账号、审批流程、安全边界要写清楚。
4. 对 Agent、MCP、LangGraph、DBHub 等技术名词，首次出现时要用中文解释它在本项目中的作用。
5. 如果写架构方案，应明确当前已实现、待实现、一阶段范围和后续阶段，不要混在一起。

## 用户可见输出

1. 前端页面、聊天回答、诊断报告、审批弹窗、错误提示必须使用中文。
2. 技术内部状态不要直接暴露给普通用户，例如 `pending_field_mapping` 应转换为“字段映射待确认”。
3. 面向管理员的接口返回可以保留结构化英文键名，但 `message`、`detail`、`summary` 等说明字段应使用中文。
4. 诊断输出应使用中文描述根因、影响范围、修复建议和下一步操作。

## 代码生成约束

1. 新增代码时优先沿用现有项目结构，不随意引入新框架。
2. 新增功能必须配套测试，至少覆盖正常路径和关键失败路径。
3. 生成文件前先确认是否已有同类文件，避免重复造目录或重复定义概念。
4. 涉及数据库密码、管理员密码、医院真实连接串的内容不得写入 Git；只提交示例配置。
5. 涉及 SQL 执行必须保持只读、安全校验、行数限制和超时控制。

## 产品工程原则

### 可用性

1. 用户日常业务操作必须在前端完成，不得把命令行作为普通用户的主要操作入口。
2. 页面必须覆盖加载中、空数据、成功、失败和权限失效状态，并给出可以执行的下一步。
3. 关键操作必须防止重复提交；失败后保留用户已经填写的内容，不得要求重新录入。
4. 前端和接口错误必须说明发生了什么、影响什么、用户下一步可以做什么，不能只显示“失败”或原始异常。

### 可维护性

1. 代码按数据请求、状态、渲染和业务动作拆分，避免大函数和跨模块隐式修改状态。
2. 优先复用已有 API、组件和状态映射；同一概念不得出现多套实现和同义状态。
3. 功能变更必须同步自动化测试、用户文档和故障定位信息。
4. 新增文件和抽象必须有清晰职责；调用方不应通过阅读内部实现才能理解输入、输出和失败语义。

### 易上手

1. 使用医院人员理解的业务语言和合理默认值，不要求用户理解 MCP、manifest、run_key 等内部术语。
2. 高频流程应在少量明确步骤内完成，并在当前操作位置提供说明和错误处理建议。
3. 命令行、原始 JSON 和技术接口只作为开发或实施排障补充，不能替代产品界面。
4. 关键按钮名称必须和操作结果保持一致；空页面应引导用户完成第一个可执行动作。

## 版本提交与推送规范

1. 每完成一批可独立验证的改动，必须先检查 Git 差异并运行与改动范围匹配的测试；验证通过后，自动暂存本批相关文件、创建 Commit，并推送到当前分支已配置的远程上游仓库，不需要再次等待用户提醒。
2. 不得提交与本批任务无关的文件，不得覆盖或回退用户已有改动，不得把密码、令牌、真实连接串、运行时数据库、日志或临时文件提交到仓库。
3. Commit 必须采用 `Conventional Commits` 规范，格式为 `<type>(可选范围): <中文主题>`。主题必须使用简洁、明确的中文描述本次改动，不使用含糊表述，例如“更新代码”“修改问题”。
4. 允许的常用类型包括：`feat`（新增功能）、`fix`（修复问题）、`refactor`（重构）、`docs`（文档）、`test`（测试）、`perf`（性能）、`chore`（工程维护）、`build`（构建）、`ci`（持续集成）和 `revert`（回退）。
5. 推荐示例：`feat: 增加指标规则 MySQL 主存储`、`fix: 修复生成 SQL 未复用会话指标`、`docs: 补充本地部署说明`。
6. 一次 Commit 只表达一个完整目的；代码、测试和直接相关文档应放在同一个 Commit 中，互不相关的改动必须拆分提交。
7. 测试失败、存在未解决冲突、远程上游未配置或推送鉴权失败时，不得强制提交或强制推送；应保留现场并向用户明确说明失败原因和当前状态。
8. 禁止使用 `git push --force`、`git push --force-with-lease` 或改写已推送历史，除非用户明确要求并确认风险。
9. 如果用户在当前任务中明确要求“暂不提交”“暂不推送”或指定其他提交方式，应以用户最新指令为准。

## 例外处理

如果某个文件必须使用英文，例如第三方标准配置、包清单、协议 schema 或库约定，应在文件顶部或相邻 README 中用中文说明该文件为什么保留英文，以及实施人员需要改哪些字段。

一句话原则：**医院人员要看得懂，实施人员要能照着做，Agent 不要用英文制造理解成本。**
