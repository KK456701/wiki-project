# 结构化会话上下文与本地模型窗口设计

## 1. 背景与目标

当前对话记忆只稳定保存上一指标、统计起止时间和少量消息元数据。用户提出“按入区时间计算”后，模型能够生成一段解释，但该选择没有进入可执行状态；后续“生成 SQL”“试运行”会重新读取 MySQL 中的本院生效口径，继续使用原来的入院时间字段。

本设计解决以下问题：

1. 用户在当前会话中明确的统计时间、时间起点、阈值、排除条件和计数方式必须被结构化保存。
2. SQL 生成、试运行、诊断、明细查看和导出必须使用同一份执行上下文快照。
3. 会话临时试算不能直接修改已审批的本院正式口径。
4. 字段映射未确认或用户表述存在关键歧义时必须停止执行，不能静默回退到旧字段。
5. Ollama 使用可配置的 16K 上下文，并保留最近 8 轮原始消息辅助本地模型理解语言，但模型历史不能作为 SQL 执行事实来源。

## 2. 已确认根因

现有 `ConversationMemory.last_rule_context()` 只从助手消息元数据中恢复：

- `rule_id`
- `rule_name`
- `stat_start_time`
- `stat_end_time`

它不保存“按入区时间”“排除 ICU”“阈值改为 48 小时”等会话口径。现有 Ollama 请求也没有传递最近原始消息，并且没有显式配置 `num_ctx`。因此问题由两部分共同造成：

- **状态缺失**：业务调整没有结构化持久化，执行链路无法使用。
- **语言上下文不足**：模型只看到当前问题和上一指标名称，无法稳定理解多轮追问。

扩大模型上下文只能改善语言理解，不能替代业务状态管理。

## 3. 核心原则

### 3.1 执行事实与语言历史分离

- 结构化会话状态是 SQL 执行的唯一会话事实来源。
- 最近原始消息只用于意图识别、指代消解和回答措辞。
- 模型输出只能产生“候选状态变更”，必须经过类型校验和业务校验后才能写入状态。

### 3.2 临时试算与正式口径分离

- `session` 范围的调整只影响当前会话，不写入 `med_index_hospital_custom`。
- 用户明确提出“以后本院都这样计算”等长期要求时，系统只生成口径差异预览，经过审批后才能成为正式本院口径。
- 新会话默认使用本院已审批生效口径，不继承其他会话的临时调整。

### 3.3 不确定时停止，不静默回退

“按入区时间”可能同时涉及：

- 统计周期用哪个时间字段筛选；
- 48 小时耗时从哪个时间点开始计算。

如果用户没有明确说明，系统应要求确认“只调整统计范围、只调整 48 小时起点，还是两者都调整”。确认前不得执行 SQL。若已经确认语义，但医院字段尚未映射，也必须停止并展示候选字段，不能继续使用 `ADMITTED_AT`。

## 4. 状态模型

在现有 SQLite 会话库中增加 `session_contexts` 表，每个会话保存一份带版本号的 JSON 状态：

```text
session_contexts
├── session_id          主键，关联 sessions
├── context_version     乐观锁版本
├── context_json        结构化会话状态
├── created_at
└── updated_at
```

应用层使用 Pydantic 契约校验，核心结构如下：

```json
{
  "schema_version": 1,
  "active_rule": {
    "rule_id": "MQSI2025_001",
    "rule_name": "患者入院48小时内转科的比例",
    "hospital_id": "hospital_001"
  },
  "stat_period": {
    "start_time": "2026-07-01 00:00:00",
    "end_time": "2026-08-01 00:00:00",
    "source_message_id": 101
  },
  "working_caliber": {
    "scope": "session",
    "overrides": [
      {
        "key": "elapsed_time_start",
        "business_value": "ward_entry_time",
        "hospital_field": null,
        "status": "pending_mapping",
        "source_message_id": 105,
        "source_text": "48小时从入区时间开始算"
      }
    ]
  },
  "pending_clarifications": [],
  "last_action": "explain_rule"
}
```

`overrides` 采用可扩展键值设计，首批支持：

- `period_time_field`：统计周期筛选时间字段；
- `elapsed_time_start`：耗时计算起点；
- `threshold_minutes`：时限阈值；
- `excluded_departments`：排除科室；
- `counting_method`：去重或多次计数方式；
- `additional_filters`：其他已确认筛选条件。

每项调整必须保留来源消息、作用范围、确认状态和医院字段映射状态，便于审计和定位。

## 5. 上下文解析与合并

每轮请求按以下步骤处理：

1. 读取结构化会话状态。
2. 读取最近 8 轮原始用户与助手消息，仅供模型理解。
3. 规则解析器先提取日期、清除命令和明确参数。
4. LLM 将自然语言转换为类型化的 `ContextDelta`，不直接修改状态。
5. 状态管理器校验指标归属、歧义、字段角色和数值范围。
6. 合并成功后写入 `session_contexts`，版本号加一。
7. 生成本次不可变的 `ExecutionContextSnapshot`。
8. 后续 SQL、试运行、明细和导出全部引用该快照。

状态优先级固定为：

```text
当前消息中已确认的明确要求
> 当前会话 working_caliber
> 本院已审批生效口径
> 公司标准
> 国标口径
```

不允许模型根据最近聊天文本绕过此优先级。

## 6. 指标切换与状态生命周期

- 同一指标追问：保留统计区间和兼容的会话调整。
- 切换指标：保留统计区间；清除不属于新指标的 `working_caliber`，并记录清除原因。
- “清除刚才的调整”“恢复本院口径”：清空当前指标的会话调整，恢复本院生效口径。
- “新会话”：创建空结构化状态，不继承上一会话临时调整。
- 正式口径审批成功：当前会话可选择刷新到新版本；已生成的执行快照仍保留原版本，保证结果可复现。

## 7. “按入区时间”场景

用户输入“按入区时间怎么算”时，系统不应直接假定含义，而是形成待确认项：

```text
你希望：
1. 统计范围按入区时间筛选；
2. 48 小时从入区时间开始计算；
3. 两者都按入区时间。
```

用户确认后：

1. 将选择写入 `working_caliber.overrides`。
2. 查询本院字段映射中是否存在 `ward_entry_time`。
3. 已映射则标记为 `ready`，后续生成 SQL 必须使用该字段。
4. 未映射则标记为 `pending_mapping`，展示候选字段并阻止 SQL 生成和试运行。
5. 用户确认医院字段后，再生成临时试算 SQL。

用户随后输入“生成 SQL 并执行”时，回复和链路必须明确显示：

```text
本次临时试算：按入区时间计算
调整范围：仅当前会话
是否修改本院正式口径：否
统计周期字段：已确认字段 / 待确认
48 小时起点字段：已确认字段 / 待确认
```

## 8. Ollama 上下文预算

项目通过配置显式控制每次 Ollama 请求，不依赖全局环境变量或模型 Modelfile：

```yaml
ollama_num_ctx: 16384
ollama_history_turns: 8
ollama_prompt_budget_tokens: 12000
ollama_output_reserve_tokens: 1200
ollama_timeout_seconds: 60
```

普通请求和流式请求都传递：

```json
{
  "options": {
    "temperature": 0.2,
    "num_ctx": 16384,
    "num_predict": 700
  }
}
```

提示词内容按以下顺序装配：

1. 系统指令；
2. 结构化会话状态摘要；
3. 当前指标必要事实和字段映射；
4. 最近 8 轮原始消息；
5. 当前用户问题；
6. 输出预留空间。

超过预算时优先删除最旧原始消息，然后压缩非必要规则说明；结构化状态、当前问题和关键字段映射不得裁剪。完整 SQL、患者明细和大段工具结果不进入模型历史，只保留必要摘要。

16K 是医院本地 4B 模型的首批默认值。32K 仅在性能测试通过后按部署机器配置启用，262K 理论上限不作为生产默认值。

## 9. 执行链路与可观测性

核心对话链路增加两个节点：

### `context_resolve` 解析会话上下文

展示：

- 当前指标；
- 统计区间；
- 读取的历史轮数；
- 结构化状态版本；
- 本轮识别出的候选变更；
- 是否存在待确认问题。

### `working_caliber_apply` 应用本次试算口径

展示：

- 每项会话调整；
- 来源消息；
- 覆盖的正式口径字段；
- 医院字段映射状态；
- 最终采用值及优先级；
- 阻断原因。

同时记录 Ollama 观测数据：配置窗口、提示词估算 token、保留历史轮数、裁剪消息数和响应耗时。链路不得记录患者明细、完整 SQL 参数中的敏感值或未经脱敏的历史数据。

## 10. 错误处理

- 状态 JSON 校验失败：保留上一有效版本，记录错误并提示重试。
- 并发更新冲突：按 `context_version` 检测，重新读取后只合并本轮 `ContextDelta`。
- 关键语义有歧义：写入 `pending_clarifications`，禁止执行。
- 医院字段未映射：标记 `pending_mapping`，禁止执行并提供候选字段。
- LLM 输出无效：退回规则解析结果，不清空既有状态。
- Ollama 不可用：结构化状态仍可保存；使用规则模板回答，并明确模型暂不可用。
- 状态存储不可用：不得假装已记住调整，应提示本轮不能安全继承口径。

## 11. 兼容与迁移

1. 保留现有 `messages` 和 JSONL 审计日志。
2. 首次读取没有 `session_contexts` 的旧会话时，从最近助手元数据迁移 `rule_id`、指标名和统计区间。
3. 旧消息中的其他自然语言条件不自动升级为可执行状态，避免错误推断；需要用户重新确认。
4. 现有 MySQL 正式规则、审批、版本和回滚流程不变。

## 12. 验收标准

必须覆盖以下自动化测试：

1. 查询指标后说“48小时从入区时间开始算”，再说“生成 SQL”，生成器收到 `elapsed_time_start=ward_entry_time`。
2. 只说“按入区时间算”时产生澄清问题，确认前不生成 SQL。
3. 入区字段未映射时不允许回退到 `ADMITTED_AT`。
4. 刷新页面并使用同一会话后，临时调整仍可恢复。
5. 新会话不继承旧会话临时口径。
6. 切换指标时清理不兼容调整，但保留统计区间。
7. “恢复本院口径”能够清除会话调整。
8. “以后本院都按入区时间”进入审批预览，不直接修改正式规则。
9. SQL 生成、试运行、明细与导出使用同一执行上下文快照。
10. Ollama 普通和流式请求都传递 `num_ctx=16384`。
11. 超过提示词预算时只裁剪旧原始消息，不丢失结构化状态。
12. 执行链路能够说明最终时间字段来自用户本轮、会话临时口径还是正式本院口径。

## 13. 本批范围外

- 不自动猜测医院“入区时间”具体对应哪个字段。
- 不让 LLM 直接发布或审批正式口径。
- 不把完整数据库结构、患者明细或全部历史 SQL 放入模型上下文。
- 不默认启用 32K 或 262K 超长上下文。
