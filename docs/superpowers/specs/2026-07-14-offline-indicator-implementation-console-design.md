# 离线指标实施控制台与安全交付架构设计

## 1. 设计状态

- 设计日期：2026-07-14
- 设计结论：已确认
- 交付方式：公司与医院通过双向离线签名包交换
- 元数据范围：医院管理员选择的白名单结构元数据
- 适用对象：医生、医务和质控人员、医院信息科、实施人员、公司规则维护人员

本文作为以下既有设计的兼容升级总设计，不另建平行系统：

- `2026-07-10-hospital-company-kb-exchange-design.md`
- `2026-07-10-indicator-generation-closed-loop-design.md`

现有“指标设计稿”升级为“指标实施控制台”；现有 `kb-exchange-v3` 和
`company-release-v2` 分别演进为医院反馈包和公司发布包，不创建含义重复的第三套包格式。

## 2. 目标

1. 让公司在不接触患者数据的前提下，根据医院白名单结构元数据设计指标适配方案。
2. 让医院在本地完成字段映射确认、SQL 生成、真实试运行、审批、生效和回退。
3. 明确 `information_schema` 只能描述数据库结构，不能自动决定分子、分母和字段语义。
4. 支持用户临时询问不同口径下的结果，同时保证当前生效口径不被隐式修改。
5. 将 Wiki 知识、术语、规则和公司 Skill 纳入可验签、可审计、可回退的离线交付流程。
6. 保持医生界面使用业务语言，技术字段、SQL 和包清单只在需要时展开。

## 3. 非目标

- 公司端不连接医院 HIS、EMR 或其他患者业务数据库。
- 公司端不接收患者明细、字段样例值、查询结果明细或数据库凭据。
- 不允许大模型直接生成任意 SQL 后跳过字段确认和安全校验。
- 公司侧结构校验不能替代医院本地真实数据试运行。
- 公司发布包不能在导入后自动覆盖医院当前生效口径。
- 首期公司 Skill 不支持任意 Python 或操作系统脚本执行。

## 4. 当前实现与需要升级的部分

### 4.1 当前已经具备

- `med_index_standard` 保存公司或国标基线，`med_index_hospital_custom` 保存本院生效口径。
- `med_metadata_table`、`med_metadata_column` 和 `med_table_relation` 保存医院本地结构元数据。
- `med_field_mapping` 保存已确认的业务字段到医院表字段映射。
- SQL 生成优先读取生效口径、字段映射、SQL 模板和参数，并执行只读安全校验。
- 指标设计稿支持自然语言解析、字段确认、SQL 生成、试运行、审批、版本和回退。
- 医院端可以导出 `kb-exchange-v3`，公司端可以导出 `company-release-v2`。

### 4.2 当前需要升级

- 元数据候选检索仍依赖设计稿预先猜测的主表，需要改为元数据优先检索。
- 第一版设计稿主要支持单表指标，需要增加已确认的多表关联配置。
- 公司发布包尚未形成医院端“验签、冲突检查、适配、审批、应用”的闭环。
- 不同假设口径缺少独立沙盒，容易与当前生效口径混淆。
- 医院反馈包需要补齐白名单结构元数据、本院口径差异和试运行反馈。
- 公司 Skill 缺少统一清单、权限声明、版本、验签和执行审计。

## 5. 总体架构

系统采用同一套产品代码、两种部署模式。

### 5.1 公司端：标准与交付中心

公司端负责：

- 维护标准指标、结构化分子分母定义和 Wiki 说明；
- 维护术语、同义词、业务字段字典和字段类型要求；
- 导入医院反馈包，在白名单元数据上生成字段映射建议；
- 编译候选 SQL 方案并完成结构、语法和只读安全校验；
- 生成带版本、校验和及数字签名的公司发布包；
- 审核医院回传的口径经验和术语候选，但不自动提升为公司标准。

公司端可以证明字段存在、类型兼容、参数完整和 SQL 只读，但不能证明真实分子、分母结果正确。

### 5.2 医院端：指标实施控制台

医院端负责：

- 选择允许导出的数据库、表和字段白名单；
- 导入并验证公司发布包的签名、版本和依赖；
- 确认业务字段对应本院哪张表、哪个字段以及表间关联；
- 使用本院生效口径和已确认映射编译候选 SQL；
- 通过 DBHub 使用只读账号在院内数据库试运行；
- 核对分子、分母、未达标记录和数据来源；
- 完成业务、技术和管理审批后发布；
- 停用或回退到历史版本；
- 导出不包含患者数据的医院反馈包。

## 6. 安全边界

### 6.1 允许进入医院反馈包

- 医院管理员选中的数据库逻辑标识、表名和字段名；
- 字段类型、是否允许为空、主外键信息和经过审核的字段注释；
- 已确认的表关联关系；
- 当前已审批且处于生效期的本院口径；
- 本院口径与标准口径的字段级差异；
- 已确认字段映射及其版本；
- 试运行状态、错误码、警告和分子分母等聚合结果；
- 包版本、来源医院、导出人、导出时间、校验和及数字签名。

### 6.2 默认禁止进入任何离院包

- 患者标识、患者明细和字段样例值；
- 查询结果明细、诊断样本和会话内容；
- 数据库账号、密码、令牌、连接串和真实网络地址；
- 未经管理员选择的数据库结构；
- 未审批草稿和已失效医院口径；
- 运行日志原文和包含患者信息的错误堆栈。

### 6.3 可选敏感内容

完整医院 SQL 会暴露内部表字段和关联方式，默认不进入反馈包。医院管理员明确授权时才可导出，且必须在清单中单独列示。默认使用结构化计算定义、字段映射和表关系替代完整 SQL。

## 7. 双向离线包

### 7.1 公司发布包

对外文件建议使用 `.medpkg`，内部继续使用 ZIP 和结构化文件，格式版本由
`company-release-v2` 演进。至少包含：

- `manifest.json`：包编号、格式版本、系统版本范围、依赖和权限清单；
- 标准指标、结构化计算定义和版本快照；
- Wiki 制度原文、医生可读说明和故障兜底内容；
- 术语、同义词和业务字段字典；
- 医院字段映射建议和推荐依据；
- SQL 计划或模板，不包含患者参数值；
- 受控 Skill 清单；
- 逐文件 SHA-256 和公司数字签名。

医院导入后只进入“待适配”状态，不能直接生效。

### 7.2 医院反馈包

对外文件建议使用 `.medfeedback`，内部由现有 `kb-exchange-v3` 演进。至少包含：

- 白名单结构元数据；
- 已确认表关联关系；
- 当前生效本院口径及与标准口径的差异；
- 已确认字段映射；
- 试运行状态、聚合结果、错误码和兼容性反馈；
- 来源医院、导出范围、审批信息、校验和及医院签名。

本院口径示例：

```text
指标：急会诊及时到位率
标准口径：10分钟内到位
本院口径：20分钟内到位
差异字段：arrive_minutes_threshold
本院版本：v1
状态：已审批、生效中
```

### 7.3 兼容策略

- 保留现有知识包和公司发布包接口，内部增加新版格式处理。
- 旧包只读兼容，不为旧包补造不存在的签名或审批信息。
- 重复导入同一包必须幂等；内容相同返回已导入记录，内容不同但包编号相同则拒绝。
- 任一文件被修改、签名无效或依赖不满足时，整个包进入隔离区，不写正式规则。

## 8. SQL 生成机制

### 8.1 生成依据

正式 SQL 必须由以下五部分共同决定：

1. 指标计算定义：分子、分母、筛选、排除、统计周期和计算公式；
2. 业务字段要求：例如申请时间、到位时间、会诊类型和患者标识；
3. 医院字段映射：业务字段对应医院数据库中的实际表字段；
4. 表关联关系：多表指标的连接字段、方向、连接类型和基数；
5. 生效参数：医院、统计区间、阈值和业务编码。

`information_schema` 只能回答“有哪些表和字段”，不能回答“哪个字段属于分子或分母”。系统可以根据字段名、注释、类型、术语和历史映射给出候选，但映射和关联必须由医院确认。

### 8.2 生成过程

```text
结构化口径
  -> 所需业务字段合同
  -> 白名单元数据候选检索
  -> 医院字段和表关联确认
  -> 类型化 SQL 计划
  -> 确定性 SQL 编译
  -> 只读与医院隔离校验
  -> 医院本地试运行
  -> 审批发布
```

模型可以负责把自然语言转成结构化草稿、解释候选映射和补充医生可读说明，但不能直接跳过类型化计划生成可发布 SQL。

### 8.3 SQL 输出组成

每个候选版本同时生成：

- 正式统计 SQL：一次运行返回分子、分母、未达标数量和指标值；
- 可解释步骤：筛选统计范围、计算中间值、筛选分子、统计分母、计算比例；
- 明细查询模板：分别查看统计范围、达标记录和未达标记录；
- 字段血缘：每个计算步骤使用的数据库、表、字段和运算关系；
- 参数清单：本院口径值、统计区间和业务编码来源。

正式统计 SQL 优先使用命名公共表表达式或等价子查询表达基础范围、分子和分母，使统计结果与明细模板复用同一计算定义，避免三套逻辑漂移。

## 9. 当前口径与口径沙盒

### 9.1 当前生效口径

用户直接说“生成 SQL”或“试运行”时，系统使用当前医院已审批且处于生效期的口径；没有有效本院口径时使用标准口径。标准记录保持不可修改，本院口径作为独立版本存在。

### 9.2 口径沙盒

用户说“如果按10分钟计算”或“排除周末后怎么算”时创建临时沙盒版本：

- 继承当前生效口径；
- 只保存用户明确修改的差异；
- 生成候选 SQL 并在医院本地试算；
- 展示当前口径与沙盒口径的参数、分子、分母和结果差异；
- 不改变当前生效版本，也不进入定时计算；
- 用户确认采用后，才转换为设计稿并进入试运行、审批和发布。

## 10. 指标实施控制台

现有“指标设计稿”升级为“指标实施控制台”。聊天仍是主要入口，控制台负责准确性、审批和审计。

### 10.1 定义指标

- 新增指标、本院口径调整或口径沙盒；
- 指标名称、定义、分子、分母、筛选、排除和统计周期；
- AI 助手可以生成初稿，但所有结构化字段可人工编辑。

### 10.2 确认取数要求

- 将技术字段转换成医生能理解的业务信息；
- 标明每个业务信息在分子、分母或公共统计范围中的作用；
- 显示必须字段、可选字段和缺失字段。

### 10.3 映射医院数据

- 从白名单元数据中选择数据库、表和字段；
- 展示候选推荐、推荐依据和置信度；
- 配置多表连接字段、连接类型和数据基数；
- 医院信息科确认后形成不可变映射版本。

### 10.4 生成和验证

- 编译候选 SQL，显示医生可读的分子分母说明和字段血缘；
- 执行只读安全检查和医院本地试运行；
- 显示分子、分母、未达标数量、结果和院内明细入口；
- 明细数据只在医院本地预览或下载，并按现有脱敏和自动清理策略处理。

### 10.5 审批和发布

- 医务或质控人员确认业务口径；
- 信息科确认字段映射、关联关系和 SQL；
- 管理员批准生效；
- 发布形成不可变快照，支持停用、版本对比和追加式回退。

## 11. 状态机

统一产品状态如下：

```text
设计中
  -> 待确认取数要求
  -> 待确认字段映射
  -> SQL待验证
  -> 试运行通过
  -> 待审批
  -> 已生效
```

附加状态包括：审批退回、试运行失败、已停用、已回退和包隔离。内部可以继续使用现有英文状态值，但 API 的 `message`、前端和执行链路必须显示中文业务名称。

关键约束：

- 修改口径、字段映射、表关联或 SQL 计划后，旧试运行结果立即失效；
- 只有当前设计稿版本的试运行通过结果可以提交审批；
- 回退通过复制历史快照创建更高版本实现，不修改旧记录；
- 包导入、适配或发布失败不得影响当前生效指标。

## 12. 受控公司 Skill

公司 Skill 是受控工作流能力，不是可任意执行的脚本。每个 Skill 必须声明：

- 唯一标识、名称、版本、适用系统版本和负责节点；
- 类型化输入、输出和错误码；
- 允许调用的本地工具与最大执行时间；
- 是否允许访问白名单元数据、Wiki、术语库或只读 DBHub；
- 是否禁止联网和文件写入；
- 配置校验和、公司签名及回退版本。

首期只支持：

- 声明式工作流节点；
- 提示词模板；
- 结构化解析器配置；
- SQL 规则和解释模板；
- 现有受控工具的权限组合。

医院管理员导入时查看权限清单并决定启用。每次执行记录 Skill 版本、输入摘要、输出摘要、耗时和失败原因，不记录患者明细。

## 13. 异常处理

前端错误必须说明发生步骤、影响范围、处理人和下一步，不直接显示孤立技术异常。

主要错误类型：

- 发布包签名无效或文件校验和不一致；
- 系统版本、规则版本或 Skill 依赖不兼容；
- 白名单元数据缺少必要表字段；
- 一个业务字段存在多个候选且未确认；
- 多表关联关系缺失或基数风险未确认；
- SQL 编译、只读安全校验或参数绑定失败；
- 试运行分母为零、结果异常或数据质量检查失败；
- 当前口径版本与发布包基础版本冲突；
- 审批角色、状态或试运行版本不满足发布条件。

所有失败保持现场和用户输入，不要求重新录入；可以回到对应步骤修正。

## 14. 审计与权限

- 医生或质控人员：定义业务口径、查看试算结果、提交审批；
- 信息科或实施人员：同步元数据、确认字段映射和表关联、处理技术异常；
- 医院管理员：选择导出白名单、导入验签、审批发布、停用和回退；
- 公司规则人员：维护标准、审核反馈、生成发布包；
- 公司人员不能通过包获得医院患者数据或远程执行医院 SQL。

导入、导出、映射、试运行、审批、发布、停用、回退和 Skill 执行均记录操作人、时间、医院、指标、版本、原因和结果。

## 15. 分阶段落地

### 第一阶段：离线包安全闭环

- 医院白名单元数据选择与预览；
- 医院反馈包补充本院口径、结构元数据和聚合反馈；
- 公司发布包医院侧验签、隔离、预览和兼容检查；
- 双方包的签名、幂等、审计和版本兼容。

### 第二阶段：元数据优先映射工作台

- 业务字段合同；
- 候选字段检索和人工确认；
- 多表关联配置与风险检查；
- 映射版本和字段血缘。

### 第三阶段：指标实施控制台闭环

- 将现有设计稿界面升级为五步控制台；
- 统一 SQL 统计、解释和明细模板；
- 口径沙盒对比；
- 试运行、审批、发布和回退联动。

### 第四阶段：公司 Skill 治理

- Skill 清单、验签和权限预览；
- 声明式节点导入与启停；
- 执行审计、兼容检查和版本回退。

## 16. 验收标准

- 医院只能导出管理员明确选择的白名单结构元数据。
- 医院反馈包包含当前生效本院口径及与标准口径的差异，不包含患者明细。
- 公司发布包在医院端完成验签和兼容检查前不能进入适配流程。
- 公司发布包导入后不能直接覆盖当前生效指标。
- `information_schema` 候选必须经过医院确认才能成为正式字段映射。
- 单表和已配置关联关系的多表指标可以生成确定性只读 SQL。
- SQL 统计值、医生解释和明细查询来自同一结构化计算定义。
- 口径沙盒可以对比不同口径结果，但不会改变线上版本。
- 试运行失败、包损坏、版本冲突和审批退回不会影响当前生效指标。
