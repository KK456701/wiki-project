# 分子分母字段与运算拆解设计

## 背景

当前回答已经能够用自然语言说明分母和分子的业务含义，并在技术详情中展示字段映射，但两部分之间仍有断层：医生看不到每一步具体用了哪些医院字段，实施人员则需要在多个表格间自行拼接“筛选、计算、计数、相除”的完整过程。

本设计在不暴露患者明细和不依赖大模型自由解释的前提下，将结构化指标定义转换为一条可核对的计算链路。

## 目标

1. 默认回答直接说明数据来自哪个数据库、哪些表。
2. 分母和分子分别列出涉及字段、字段含义、自然语言规则和运算关系。
3. 使用编号步骤串起筛选范围、派生计算、计数和最终比例。
4. 医院生效口径必须落到具体条件和字段，并说明只影响分子、分母或两者。
5. 展示内容与实际 SQL 共用结构化计算定义和字段映射，禁止根据字段名称猜测。

## 默认展示结构

生成 SQL 和试运行回答都在“当前采用什么规则”之后展示以下内容。

### 1. 数据来源

使用独立信息行显示：

- 数据库名称，例如 `hospital_demo_data`。
- 涉及的数据表，例如 `consult_record`。
- 每张表在本指标中的用途，例如“提供急会诊申请、到位时间和会诊类型”。
- 数据库类型，例如 MySQL。

存在多张表时逐表列出，不得只显示主表。字段映射或数据库名称缺失时，明确标记“尚未配置”，不得推测。

### 2. 一句话计算说明

按固定顺序生成一段可以直接复述的说明：

> 先从 hospital_demo_data 数据库的 consult_record 表中，筛出本院在统计期内的急会诊记录作为分母；再用每条记录的到位时间减去申请时间，0 至本院规定的 20 分钟内记为及时到位，所有及时到位记录相加得到分子；最后用分子除以分母并乘以 100%。

说明中的表、条件和阈值全部来自结构化配置。

### 3. 分子与分母拆解

使用一张四列表格展示：

| 统计项 | 涉及的医院数据 | 系统怎样计算 | 运算关系 |
| --- | --- | --- | --- |
| 分母 | 医院、会诊类型、申请时间，并附真实表字段 | 先按医院、会诊类型和统计区间筛选，再对符合条件的记录计数 | `分母 = 符合全部范围条件的记录数` |
| 分子 | 继承分母，再使用申请时间和到位时间，并附真实表字段 | 到位时间减申请时间得到分钟数；在阈值内记 1，否则记 0，最后相加 | `分子 = SUM(是否及时：是=1，否=0)` |
| 指标结果 | 分子和分母 | 分母不为 0 时相除并乘以 100，保留两位小数 | `指标值 = 分子 / 分母 x 100%` |
| 样本数 | 与分母相同 | 用于说明本次统计覆盖的记录量 | `样本数 = 分母` |

字段展示采用“中文含义 + 医院数据库位置”，例如：

- 急会诊申请时间：`consult_record.request_time`
- 急会诊到位时间：`consult_record.arrive_time`

医生可以先读中文，信息科仍可直接核对字段。

### 4. 详细执行步骤

以编号列表展示确定性的执行过程：

1. **筛选统计范围**：按医院、统计时间和分母业务条件筛选原始记录。
2. **计算派生值**：需要时间差等派生字段时，说明参与字段和加减关系；MySQL 时间差同时标注 `TIMESTAMPDIFF` 的含义。
3. **统计分母**：说明按记录计数还是按某字段去重计数。
4. **统计分子**：说明从分母中追加哪些条件，以及布尔条件如何转换为 1/0 后求和。
5. **计算指标**：说明除法、乘以 100、保留位数和分母为 0 时的处理。

没有派生字段的指标跳过第 2 步并重新编号，不生成空步骤。

## 医院口径展示

医院口径仍以当前生效版本为准，并在拆解中落到实际条件：

- “到位时限 20 分钟”关联申请时间和到位时间。
- 标准值 10 分钟作为对照信息，不参与本次计算。
- 明确写出该差异只改变分子，不改变分母。

其他医院定制条件按其所在阶段说明影响范围，不统一假设只影响分子。

## 技术详情边界

默认展示新增字段与运算拆解，但不展示完整 SQL、SQL ID、安全校验细节和原始配置 JSON。这些信息继续放在“查看技术详情”折叠区。

现有“分母如何取数”“分子如何从分母中筛选”技术表格继续保留，用于排查规则来源、条件来源和本院字段映射；默认区不重复展示这些内部列。

## 生成方式

说明必须由以下结构化数据确定性生成：

1. `CalculationDefinition` 提供范围条件、分母、分子、派生字段、聚合方式和比例运算。
2. 指标字段血缘提供业务字段、医院物理字段、真实表和数据库。
3. 本院生效口径提供本次参数、标准参数、版本和影响范围。
4. SQL 生成结果提供数据库类型。

第一阶段支持现有计算定义中的运算：等于、不等于、非空、左闭右开区间、闭区间、分钟时间差、按行计数、按字段去重计数和百分比。遇到未知运算时显示“该运算尚未配置通俗解释”，不得自行推断。

## 异常与降级

1. 缺少数据库或表映射时显示“数据来源尚未配置完整”。
2. 缺少字段映射时保留中文字段含义，并将数据库位置标记为“尚未映射”。
3. 缺少结构化计算定义时保留原有配置不完整提示，不生成猜测性的运算步骤。
4. 分母为 0 时显示“本期没有符合统计范围的数据，暂不能计算该指标”，不得显示为 0%。
5. 分子大于分母时继续标记结果异常。

## 验收标准

1. 急会诊指标默认回答能看到 `hospital_demo_data`、`consult_record` 及其用途。
2. 分母明确列出医院、会诊类型、申请时间及对应医院字段。
3. 分子明确列出申请时间、到位时间，并说明到位时间减申请时间、阈值判断和求和。
4. 默认回答显示 `分子 / 分母 x 100%`，并说明分母为 0 的处理。
5. 本院 20 分钟与标准 10 分钟的差异关联到具体字段，且明确只影响分子。
6. 四个已落地指标都能根据各自配置生成拆解；没有时间差的指标不显示无关步骤。
7. 技术详情中的完整 SQL 和规则来源仍可查看。
8. 自动化测试覆盖按行计数、去重计数、时间差、有无医院口径、缺失映射和分母为 0。
