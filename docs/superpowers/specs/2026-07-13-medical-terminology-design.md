# 35 个核心制度指标医学术语库设计

## 1. 背景与目标

当前系统具备指标规则名称、描述的 MySQL 模糊检索，以及 Wiki `aliases` 的精确、子串和字符重叠匹配，但缺少统一、可审核的医学术语标准化层。LLM 可能理解“上感”和“感冒”等相近表达，但这种理解不可审计，也不能直接作为 SQL 条件依据。

本次建设覆盖现有 35 个核心制度指标中出现的完整业务术语范围，同时支持两类能力：

1. 将简称、口语和相关表达标准化，用于意图理解和指标召回。
2. 将已审核术语转换为本院确认的编码或数据库值，用于安全的参数化 SQL。

“完整”是指覆盖 35 个指标名称、定义、分子、分母、筛选、剔除和统计维度涉及的术语，不宣称覆盖整个医学知识体系。

## 2. 设计原则

- MySQL 保存可执行、可审核、可版本化的权威术语数据。
- Wiki 保存定义、来源、使用场景和易混淆说明，并作为导入来源和故障兜底。
- 医院映射优先于公司标准映射，但医院不能修改公司标准概念本身。
- 用于召回的关系与用于 SQL 的关系分开控制。
- LLM 只能生成待审核候选词，不能直接发布或进入正式 SQL。
- SQL 只使用参数绑定后的本院编码或值，不拼接用户原文。
- 歧义、缺少本院映射或存在禁止替换关系时，系统必须停止自动替换并给出明确提示。

## 3. 术语分类

### 3.1 概念类型

- `indicator`：指标名称和指标口径概念。
- `diagnosis`：诊断和疾病概念。
- `department`：科室和诊疗单元。
- `staff_role`：人员、职级和岗位。
- `procedure`：手术、操作和诊疗行为。
- `time_window`：时限、周期和工作日概念。
- `status`：完成、成功、有效、非计划等状态。
- `data_value`：医院数据库中的枚举值或编码含义。
- `business_concept`：会诊、查房、危急值、抢救、用血等业务概念。

### 3.2 关系类型

- `exact`：完全同义。
- `abbreviation`：简称或缩写。
- `colloquial`：业务口语表达。
- `related`：相关但不等价，只能扩大检索。
- `value_mapping`：标准概念与本院编码或数据值的映射。
- `forbidden`：容易混淆但不允许替换。

关系类型不直接决定 SQL 权限。每条别名还必须具有独立的 `retrieval_enabled` 和 `sql_safe` 标志，并通过审核后才能生效。

## 4. 数据模型

### 4.1 标准概念表 `med_term_concept`

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `concept_code` | 稳定的概念编码，唯一 |
| `canonical_name` | 标准名称 |
| `concept_type` | 概念类型 |
| `definition` | 概念定义 |
| `standard_code` | ICD 或其他标准编码，可为空 |
| `source_level` | `national` 或 `company` |
| `source_reference` | 来源文档和定位信息 |
| `version` | 概念版本 |
| `status` | `draft`、`active`、`inactive` |
| `created_at` / `updated_at` | 审计时间 |

### 4.2 同义词关系表 `med_term_alias`

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `concept_code` | 关联标准概念 |
| `alias_text` | 别名、简称或相关表达 |
| `relation_type` | 关系类型 |
| `retrieval_enabled` | 是否可用于理解和检索 |
| `sql_safe` | 是否允许进入 SQL 值映射阶段 |
| `ambiguity_group` | 歧义分组，可为空 |
| `source_reference` | 来源和关联指标 |
| `approval_status` | `pending`、`approved`、`rejected` |
| `version` | 关系版本 |
| `created_by` / `approved_by` | 操作和审核人 |
| `created_at` / `approved_at` | 审计时间 |

同一别名可以指向多个概念，但必须标记歧义。未标记歧义的已生效别名不能在同一作用域内形成冲突映射。

### 4.3 医院本地映射表 `med_hospital_term_mapping`

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `hospital_id` | 医院编号 |
| `concept_code` | 标准概念 |
| `code_system` | ICD、院内诊断、科室、职级或业务枚举体系 |
| `local_code` | 本院编码，可为空 |
| `local_name` | 本院标准名称 |
| `local_value` | 数据库实际保存值 |
| `approval_status` | 审核状态 |
| `effective_from` / `effective_to` | 生效区间 |
| `version` | 不可变版本号 |
| `created_by` / `approved_by` | 操作和审核人 |
| `created_at` / `approved_at` | 审计时间 |

本表只描述概念对应的本院编码和值。具体查询哪个表和字段，继续由现有指标字段映射负责，避免术语库和元数据职责混淆。

### 4.4 版本发布

术语发布使用不可变版本。每次发布记录版本号、内容校验和、变更摘要、发布人和发布时间。回退通过原子切换当前生效版本完成，不修改或删除历史数据。

## 5. Wiki 组织与同步

- 在 Wiki 中为标准概念生成可读术语页，包含定义、别名、关系、易混淆词、来源和关联指标。
- 35 个指标页面通过 `concept_code` 引用术语，不复制可执行映射。
- 发布成功后同步生成 Wiki 页面和检索索引；MySQL 仍是运行时执行依据。
- MySQL 暂时不可用时，Wiki 只能提供只读召回和解释，不能提供未经数据库确认的 SQL 值映射。
- 公司知识包包含术语版本、结构化术语数据、Wiki 页面和校验清单。
- 医院回收包只包含本院映射、候选词、使用反馈和冲突报告，不包含患者明细。

## 6. 术语整理范围

对每个指标逐项提取并审核：

- 指标名称、简称、口语表达和历史叫法；
- 统计对象、业务动作、结果状态；
- 分子、分母、筛选和剔除条件；
- 时间窗口、统计周期和比较关系；
- 科室、人员职级、手术等级和诊疗状态；
- 诊断、会诊、抢救、危急值、用血等医学业务概念；
- 数据库可能使用的编码、中文值、英文值和数字值；
- 易混淆但口径不同的术语。

每个候选词必须保存来源指标和来源段落。自动提取和 LLM 只负责生成候选项，最终关系类型、SQL 安全等级和歧义状态由人工审核。

## 7. 医学安全边界

文字相近不能自动视为医学等价。例如：

- “上感”和“感冒”可用于互相召回，但只有本院诊断字典确认映射到同一编码后才能用于 SQL。
- “急会诊”和“急诊会诊”可能分别表示紧急程度和就诊场景，默认不能直接替换。
- “抢救成功”不等于“治愈”。
- “再次住院”不等于“非计划再次住院”。
- “危急值”不等于一般异常值。

当一个表达命中多个概念，系统返回候选概念和差异说明，由用户确认后继续。找不到本院编码或值映射时，可以解释和召回指标，但不得生成声称可执行的 SQL。

## 8. 运行链路

```text
用户问题
  -> 术语识别
  -> 本院术语映射优先匹配
  -> 公司/国标术语匹配
  -> 歧义与安全等级检查
  -> 生成标准化检索语句
  -> 检索指标规则
  -> 读取指标字段映射
  -> 仅使用已审核的 SQL 安全本院值生成参数化 SQL
```

执行链路增加“术语标准化”节点。摘要展示识别数量、标准概念和是否存在歧义；详情展示原始表达、标准概念、关系类型、来源、版本、医院映射以及是否用于 SQL。节点不记录患者明细。

本地术语匹配不调用 LLM，目标耗时不超过 100ms。LLM 只参与意图理解和候选术语建议。

## 9. 维护、审核和实时更新

### 9.1 公司流程

1. 从 35 个指标语料提取候选术语。
2. 生成关系类型、来源和安全等级建议，全部进入待审核状态。
3. 管理员审核歧义、禁止替换和 SQL 安全属性。
4. 发布新的不可变术语版本。
5. 原子切换 MySQL 生效版本并更新 Wiki 页面和索引。
6. 清理术语缓存，新请求立即使用新版本，不要求重启服务。

### 9.2 医院流程

医院可以增加本院简称、诊断编码、科室编码、人员职级和值映射，但不能更改公司标准概念。医院映射同样需要审核、生效时间和版本记录。

### 9.3 回退与合并

- 回退只切换生效版本，历史发布不可覆盖。
- 医院映射优先，公司标准兜底。
- 公司接收医院回收包时展示新增、冲突、歧义和 SQL 安全等级变化。
- 合并后的候选词仍需公司审核，不能自动进入下一个正式版本。

## 10. 前端工作台

在“数据库与元数据”下增加“医学术语库”：

- 概念列表：标准名称、类型、关联指标、版本和状态；
- 同义词管理：关系类型、检索权限、SQL 权限和来源；
- 本院映射：院内编码、数据库值和生效时间；
- 待审核：差异对比、批量审批和拒绝；
- 版本管理：变更查看、发布和回退；
- 识别测试：输入自然语言，展示标准化结果和预计 SQL 参数，但不执行 SQL。

高风险操作使用明确的确认对话框。歧义、缺少映射和禁止替换必须使用通俗中文说明，不只显示内部错误码。

## 11. 验收标准

- 35 个指标均建立术语关联，公式和筛选条件中的关键概念无未分类遗漏。
- 标准名称和已审核别名能够召回正确指标。
- 歧义词不会自动选择指标或生成可执行 SQL。
- `related` 和 `forbidden` 关系不能进入 SQL 替换。
- SQL 只使用已审核、`sql_safe=true` 且具有有效医院映射的术语。
- SQL 使用参数绑定，用户输入不直接拼接。
- 缺少本院映射时明确指出缺失概念和所需映射。
- 执行链路可追溯原词、标准词、来源、版本和 SQL 使用情况。
- 发布、回退、公司下发包和医院回收包具备自动化测试。
- 术语匹配在本地完成，目标耗时不超过 100ms。

## 12. 实施批次

### 第一批：术语数据基础

新增术语表和版本机制，提取 35 个指标完整候选术语，建立审核状态、冲突检查和 Wiki 术语页。

### 第二批：运行链路接入

接入意图识别、规则检索和 SQL 参数映射，增加歧义确认、缺映射阻断和“术语标准化”执行节点。

### 第三批：维护工作台与知识包

增加术语管理、审批、识别测试、版本发布与回退，并扩展公司知识包和医院回收包。

## 13. 非目标

- 不在本期建设覆盖全部医学领域的通用知识图谱。
- 不让 LLM 自动审批医学等价关系或直接写入正式映射。
- 不用术语库取代指标口径表、元数据字段映射或 ICD 原始字典。
- 不把患者诊断明细写入 Wiki、术语包或执行链路。
