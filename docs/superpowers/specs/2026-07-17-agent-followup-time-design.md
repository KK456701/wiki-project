# Agent 追问时间范围修复设计

## 问题

二轮对话中，用户复制上一轮的两个时间示例并以“这个”指向最后一个示例时，Planner 只看到本轮文本与结构化规则编号，看不到最近对话。4B 模型随后把歧义项输出为字符串数组，未满足 `SemanticAmbiguity` 对象合约，导致两次计划校验均失败并返回泛化错误。

## 方案

采用服务端确定性归一化与有界历史补充的组合方案：

1. 会话记忆向 `AgentRunState` 提供已经压缩过的最近对话，Planner 将其作为只读语义背景，不改变结构化状态的权威性。
2. 当本轮同时含有“或/或者”、结尾为“这个/后者/第二个”，且最后一个选项包含明确时间表达时，将 Planner 的本轮输入归一化为最后一个时间范围的“结果”查询。
3. 兼容 4B 常见的 `semantic_ambiguities: ["说明"]` 输出，将字符串转换为 `{field, description}`，避免因容器形状错误直接终止。
4. 不扩展工具、权限或 SQL 执行边界；时间范围仍由 `TimeRangeResolver` 确定性解析。

## 验收

- Planner 能看到最近对话，但原始历史不能覆盖当前 `rule_id` 和统计周期。
- `2026年1月到3月”或“从6月1日至今”这个` 被归一化为 `从6月1日至今的结果`。
- 字符串形式的语义歧义可通过计划合约校验。
- 缺少可确定时间范围时仍返回用户澄清，不猜测日期。
- README 描述当前 `Planner → Compiler/Validator → Controller → Executor → Verifier` 架构及四个可选模型。
- `agent.md` 明确每批改动必须同步 README、提交并推送。
