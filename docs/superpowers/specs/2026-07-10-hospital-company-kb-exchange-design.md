# 医院与公司知识交换架构设计

## 1. 目标

在保留医院口径版本恢复和知识库打包回收能力的前提下，明确医院端与公司端的数据边界：

- 医院端 MySQL 是本院生效口径、审批版本和字段映射的唯一事实源。
- 公司端 MySQL 是回收包、候选规则、公司标准版本和发布版本的唯一事实源。
- Wiki 保存制度原文、可读说明、导入材料和数据库故障时的只读兜底，不参与审批后口径的实时写入。
- 医院与公司数据库不直连，只交换经过白名单过滤和校验的 ZIP 知识包。

本批继续只覆盖 `MQSI2025_001`、`MQSI2025_005`、`MQSI2025_014` 和 `MQSI2025_035`。

## 2. 部署边界

每家医院部署独立的 `wiki_agent_runtime`。该数据库不保存患者业务明细，只保存本院 Agent 运行所需的规则、审批版本、字段映射、SQL 记录、诊断报告和执行链路。医院现有 HIS/EMR 数据库继续通过 DBHub 只读访问。

公司另行部署 `wiki_company_kb`。公司数据库不连接医院业务库，不接收患者明细，仅接收医院主动导出的知识包。

本地演示环境允许两个数据库运行在同一个 MySQL 实例中，但必须使用不同 schema 和不同连接配置；正式医院部署不要求连接公司数据库，公司回收服务也不要求连接医院数据库。

## 3. 医院版本恢复

医院当前生效口径保存在 `med_index_hospital_custom`，不可变历史保存在 `med_index_hospital_custom_version`。

恢复历史版本时禁止修改或删除旧记录。系统复制指定快照，创建版本号更大的新版本，并记录 `source_version` 与 `change_type=restore`。事务提交后，下一次规则查询直接读取新版本，不依赖 Wiki 索引或服务重启。

## 4. 医院知识包

`GET /api/kb/export?hospital_id=...` 必须从医院 MySQL 读取：

- 当前已审批、启用且处于生效期的医院口径差异；
- 对应国标版本和医院口径版本；
- 已确认字段语义映射；
- 包格式、导出时间、来源医院、条目数量和文件校验和。

知识包格式升级为 `kb-exchange-v2`。包内使用 YAML/JSON 作为可移植表示，但这些文件只是某一时点的交换快照，不是医院运行时事实源。

知识包禁止包含：

- 患者记录、统计结果明细和诊断样本；
- 数据库密码、令牌和连接串；
- 运行日志和会话内容；
- 未审批或已失效的医院口径。

字段映射可以包含业务字段、表名、列名和数据类型，以支持公司识别产品适配模式，但不得包含字段值样本。

## 5. 公司知识中心

公司数据库包含以下职责：

- `company_standard_rule`：当前公司标准投影；
- `company_standard_rule_version`：公司标准不可变历史；
- `company_kb_package`：医院上传包及校验状态；
- `company_kb_package_item`：包内差异项与审核状态；
- `company_rule_candidate`：被采纳为公司候选的医院经验；
- `company_release` 与 `company_release_item`：公司发布版本及其规则快照。

上传流程必须先校验 ZIP 路径、文件类型、格式版本、医院编号和文件校验和，再将清单与条目写入公司 MySQL 暂存区。上传不会修改公司当前标准。

管理员选择“采纳为公司候选”时创建候选记录；选择“仅保留医院本地”或“拒绝”时只记录审核决定。候选规则必须再次进入公司发布流程，不能因单家医院上传而自动成为所有医院的标准。

## 6. 公司版本发布与下发

公司管理员从已审核候选创建草稿发布版本。发布时在一个事务内：

1. 创建公司标准新版本快照；
2. 更新公司当前标准投影；
3. 固化发布条目；
4. 将发布状态改为 `published`。

已发布版本可导出为 `company-release-v1` ZIP，供医院离线导入。医院导入公司发布包不直接覆盖医院生效口径：先更新国标/公司实现基线，再重新合成本院差异；发生基础版本冲突时进入人工审批。

本批实现公司发布包的生成，不自动在医院侧应用，避免绕过现有医院审批边界。

## 7. 兼容与故障处理

- 现有 `/api/kb/merge/*` 路径保留，返回结构尽量兼容前端。
- `kb-export-v1` 仅作为明确标记的旧包兼容输入；新导出统一生成 `kb-exchange-v2`。
- 医院 MySQL 不可用时导出失败，不从 Wiki 伪造当前医院版本。
- 公司 MySQL 写入失败时上传、审批和发布全部失败关闭；原始上传包不得直接改写 Wiki。
- 文件报告可以作为下载或人工阅读副本，但 API 查询和状态流转以公司 MySQL 为准。

## 8. 验收标准

- 恢复历史口径后，医院导出包包含新生成的恢复版本。
- 医院导出包只包含 MySQL 当前有效口径和确认字段映射。
- 任一包内文件被修改后，公司上传因校验和不一致而失败。
- 公司上传后可从 MySQL 查询报告，重启服务后状态不丢失。
- 采纳条目只进入候选库，不立即改变公司标准。
- 发布公司版本后可下载固定内容的发布 ZIP。
- 全量自动化测试通过，四指标 SQL 试运行行为不回归。
