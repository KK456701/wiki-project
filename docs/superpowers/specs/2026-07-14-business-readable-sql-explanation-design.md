# 业务可读 SQL 与试运行解释设计

## 背景

当前对话中的 SQL 生成结果主要面向研发人员，直接展示字段映射、参数和 SQL。医生和医院实施人员能够看到结果，但难以快速判断分子、分母为什么这样定义，医院实际采用了哪一版口径，以及试运行结果为什么是某个百分比。

本次改造在不暴露患者明细的前提下，让 SQL 生成和试运行结果同时满足业务核对、实施核对和技术审计。

## 目标

1. SQL 生成后先解释业务口径，再展示技术 SQL。
2. 分子、分母、数据表、业务字段和统计时间使用表格清晰展示。
3. 解释必须读取当前医院的生效口径，不能硬编码国标或公司默认参数。
4. 试运行一次返回指标值、分子数和分母数，并解释计算过程。
5. 不返回患者姓名、病历号、会诊明细或绑定参数后的完整 SQL。

## 不在范围内

- 不增加患者级明细查询和下载。
- 不改变医院口径审批、生效和版本回退规则。
- 不引入新的数据库或报表引擎。
- 不改造本批以外的诊断报告页面。

## 方案选择

采用“扩展单次聚合 SQL + 确定性业务解释器”。生成 SQL 同时返回 `index_value`、`numerator_count` 和 `denominator_count`，试运行器保留这些聚合值，回答层结合本院生效口径、SQL 规格和字段映射生成业务说明。

不采用仅修改回答模板的方式，因为现有结果无法可靠反推出分子数量；也不采用额外执行分子和分母查询的方式，避免查询次数增加和多次查询之间的数据不一致。

## 业务展示

### SQL 生成结果

回答按以下顺序展示：

1. **当前采用口径**：指标名称、生效层级、医院定制项和标准值差异。
2. **怎么算**：用自然语言说明统计对象、分母、分子和公式。
3. **从哪里取数**：数据库、主表以及业务字段到医院字段的映射。
4. **本次参数**：医院、统计区间、业务值和阈值。
5. **技术 SQL**：放在业务说明之后，供实施人员和研发核查。

示例：

| 计算项 | 业务解释 | 医院取数条件 |
|---|---|---|
| 分母 | 统计期内本院全部急会诊次数 | 会诊类型为急会诊 |
| 分子 | 分母中及时到位的急会诊次数 | 到位时间减申请时间为0至20分钟 |
| 公式 | 及时到位次数除以急会诊总次数 | 分子 / 分母 x 100% |

### 试运行结果

成功时展示统计区间、数据来源、口径来源和以下聚合结果：

| 统计项 | 数量 | 说明 |
|---|---:|---|
| 分母 | 10 | 符合本院急会诊口径的总次数 |
| 分子 | 8 | 其中0至20分钟内到位的次数 |
| 未进入分子 | 2 | 分母减分子，仅作解释 |
| 指标结果 | 80% | 8 / 10 x 100% |

回答正文同步生成一句结论：“本期共有10次急会诊进入分母，其中8次在本院规定的20分钟内到位，因此 8 / 10 x 100% = 80%。”

分母为0时不把0%解释为业务结果，而是显示“本期没有符合分母条件的数据，指标暂不可计算”。

## 数据契约

### SQL 聚合列

比例类指标的生成 SQL 必须返回：

- `index_value`：百分比结果。
- `numerator_count`：进入分子的去重业务对象数或业务记录数。
- `denominator_count`：进入分母的去重业务对象数或业务记录数。
- `sample_count`：为兼容现有监控和诊断接口，继续返回并等于 `denominator_count`。

计数类指标返回 `numerator_count`、`denominator_count` 和 `sample_count`，三者按该指标的计数定义保持一致。

### 试运行结果

`run_sql_trial` 在原有字段基础上增加：

- `numerator_count`
- `denominator_count`
- `source`
- `stat_start`
- `stat_end`

运行日志继续保存聚合结果和运行状态，不保存患者明细。现有表结构若只保存指标值，则分子分母先随本次响应和执行链路输出；是否持久化为独立日志字段另行设计，不在本批扩表。

### 解释上下文

业务解释器接收：

- 指标名称和编码。
- 生效口径及 `effective_level`、`overridden_fields`。
- SQL 规格中的分子、分母名称和逻辑。
- 医院字段映射、数据库名、主表和数据库类型。
- 实际执行参数和统计区间。
- 试运行聚合结果。

解释器只做确定性格式化，不调用 LLM，不自行猜测字段含义或口径。

## 本院口径一致性

1. SQL 参数和文字解释必须来自同一个 `effective_rule` 与生成结果 `params`。
2. 医院已审批且处于生效期的参数优先用于本院生效口径；未配置医院差异时使用标准口径。
3. 回答明确区分“本院生效口径”和“标准口径”，并列出发生变化的字段。
4. 若生效口径、SQL 参数和解释材料不一致，停止输出业务结论并提示检查口径版本，不生成猜测性说明。
5. 执行链路记录口径层级、版本和聚合数量，但不记录患者明细或绑定后的 SQL。

## 组件边界

### SQL 生成层

负责生成分子和分母聚合列。四个首批指标的标准 SQL 模板和指标设计稿 SQL 计划使用相同列名。

### SQL 试运行层

负责执行一次只读查询并读取聚合列，不负责生成业务文案。兼容旧 SQL：缺少新增列时保留原结果，但回答明确提示“旧版 SQL 未返回分子分母，暂无法展开计算过程”。

### 回答格式化层

新增独立格式化函数，分别生成 SQL 业务说明和试运行结果说明。`graph.py` 只负责收集上下文并调用格式化函数，避免继续堆叠大段字符串拼接。

### 前端

扩展现有对话安全渲染器，使其支持本功能需要的 Markdown 表格、标题、段落、行内代码和 SQL 代码块。渲染前必须先转义原始 HTML，不引入允许任意 HTML 的解析模式。业务表格在 SQL 代码块之前展示，无需新增业务页面，也不显示患者级展开按钮。

## 异常处理

- 分母为0：显示无样本，不计算业务百分比。
- 分子大于分母：标记结果异常，提示检查 SQL 或口径，不生成正常结论。
- 缺少分子分母列：兼容展示指标值，并提示重新生成新版 SQL。
- 字段映射缺失：维持现有阻断逻辑，不输出可执行 SQL。
- 医院口径缺少版本或来源：显示“口径信息不完整”，不声称采用本院定制。
- 试运行失败：展示数据库、主表、统计区间和错误定位建议，不展示伪造数量。

## 测试与验收

1. 四个首批指标生成 SQL 均包含分子数和分母数。
2. 急会诊示例返回分子8、分母10时，回答包含 `8 / 10 x 100% = 80%`。
3. `hospital_001` 的20分钟定制口径同时出现在 SQL 参数、业务说明和试运行解释中。
4. 无医院定制时正确显示标准口径，不出现“本院定制”字样。
5. 分母为0、旧 SQL 缺列、分子大于分母和执行失败均有明确提示。
6. 回答包含数据库、主表、业务字段和医院字段映射表。
7. 回答和执行链路不包含患者明细及绑定后的 SQL。
8. 现有 SQL 安全校验、监控、诊断和指标设计稿测试保持通过。
