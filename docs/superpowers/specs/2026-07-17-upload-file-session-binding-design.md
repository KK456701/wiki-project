# Excel 上传文件会话关联设计

## 目标

用户上传 Excel 后，即使后续只说“分析刚上传的文件”，Agent 也能确定性定位刚才上传的 `file_key`，不依赖模型从界面气泡猜测文件编号。

## 根因

当前上传成功消息只由前端 `createMessage()` 渲染在浏览器 DOM 中，没有作为会话消息发送给后端。随后自动填入的问题也不包含 `file_key`。确定性分发只能从当前问题和服务端最近会话中查找文件编号，因此返回“请先上传需要分析的 Excel 文件”。

## 方案

采用显式附件字段并保存到结构化会话状态：

1. 前端在上传成功后保存最近的 `file_key`，调用聊天接口时通过独立字段传递。
2. `AgentChatRequest` 新增可选 `file_key`，流式和非流式入口统一传给服务层。
3. `AgentRunState` 新增 `current_upload_file_key`。服务层打开会话后，如果本次请求携带文件编号，就更新该字段；正常会话保存时沿用现有结构化状态持久化。
4. 确定性分发分析上传文件时，优先使用结构化状态中的 `current_upload_file_key`，再回退到当前问题和历史文本中的“文件编号”。
5. 新会话清除前端最近上传编号；同一会话后续追问继续复用服务端保存的编号，上传新文件会覆盖旧编号。

## 安全边界

- `file_key` 只接受项目现有安全文件名字符和长度。
- Excel 工具继续校验文件存在，并要求文件名以当前 `hospital_id` 开头，不能借此访问其他医院文件。
- 模型不接收 Excel 二进制内容，也不负责选择文件；只有受控工具读取上传目录。
- Trace 可以显示安全文件编号，但不记录患者行级数据。

## 错误处理

- 请求没有 `file_key`、会话状态也没有历史附件时，保持当前明确提示“请先上传需要分析的 Excel 文件”。
- 文件不存在时返回 `UPLOAD_NOT_FOUND`。
- 文件属于其他医院时返回 `UPLOAD_ACCESS_DENIED`。

## 验收标准

- 前端聊天 payload 能携带上传接口返回的 `file_key`。
- 流式与非流式 API 均把 `file_key` 写入当前运行状态。
- “帮我分析刚上传的指标文件”能生成 `analyze_uploaded_indicators` 调用，并使用结构化状态中的文件编号。
- 同一会话后续追问可以复用最近上传文件；新上传文件覆盖旧文件。
- 现有文本文件编号解析继续可用。
- 自动化测试覆盖前端 payload、API/服务传递和确定性分发优先级。
