# 五页面业务工作台设计

## 1. 目标

按照需求规格说明书第 6 章，将当前以聊天页和管理弹窗为主的前端重组为五个可持续使用的业务页面：

1. AI 指标自助配置工作台；
2. 智能异常排查中心；
3. 口径规则管理后台；
4. 元数据与结构监控看板；
5. 指标运算监控面板。

聊天不再作为唯一主界面，而是变为所有业务页面都可调用的辅助入口。第六批优先重组已有能力，再补齐需求书要求但当前缺少的列表、报告、对比、导出和状态跟踪接口。

首批验收范围继续限定为四个已迁移指标：

- `MQSI2025_001` 患者入院 48 小时内转科的比例；
- `MQSI2025_005` 急会诊及时到位率；
- `MQSI2025_014` 急危重症患者抢救成功率；
- `MQSI2025_035` 术中自体血回输率。

## 2. 设计原则

### 2.1 业务页面优先

- 登录成功后的第一屏是实际工作台，不增加营销页或功能介绍页。
- 高频业务操作必须在对应页面完成，不要求普通用户使用命令行、原始 JSON 或接口文档。
- 原有弹窗只保留给短流程、二次确认和节点详情，不再承载完整业务页面。
- 每个上线页面必须同时具备加载、空数据、成功、失败和权限失效状态，不发布空白占位页。

### 2.2 渐进式重构

- 保留 FastAPI 静态资源和原生 HTML/CSS/JavaScript，不在本批引入前端框架。
- 新增独立工作台样式和脚本，逐步从 `web/index.html` 中迁移业务模块。
- 已有 API、Trace、管理员登录和医院隔离逻辑继续复用。
- 每迁移一个页面，原弹窗入口可暂时兼容，但页面成为新的主入口；页面验收完成后再移除重复入口。

### 2.3 数据安全

- 页面只展示聚合指标、规则版本、元数据结构、诊断结论和审计信息。
- 不展示患者明细、绑定后的含值 SQL、数据库密码或令牌。
- 所有医院端请求显式传递当前 `hospital_id`，后端继续校验数据归属。
- 写操作继续失败关闭：MySQL 不可用时不得写入 Wiki 兜底。

## 3. 总体信息架构

### 3.1 桌面端

工作台使用三段式结构：

- 左侧导航：五个业务页面，固定宽度约 `224px`；
- 中央主工作区：页面标题、筛选、列表、详情和业务操作；
- 右侧 AI 助手抽屉：默认收起，可在任意页面打开，保留当前会话。

顶部全局栏只保留跨页面信息：

- 当前系统名称；
- 当前医院；
- 当前用户身份；
- 系统健康状态入口；
- 恢复中心入口；
- AI 助手开关。

不再把指标监控、MCP、指标设计稿和审批都堆成顶部按钮。

### 3.2 移动端

- 左侧导航折叠为顶部“当前页面”选择器；
- 主内容改为单列，列表和详情上下排列；
- AI 助手使用全屏抽屉；
- 表格在窄屏转换为字段列表，不依赖横向滚动完成关键操作；
- 所有按钮和标题保持可见，不与底部输入区重叠。

### 3.3 页面路由

继续使用单页应用，通过 URL Hash 保存页面状态：

```text
#/indicator-config
#/diagnosis
#/caliber-rules
#/metadata
#/monitoring
```

刷新页面后恢复当前业务页面。未知 Hash 回到 `#/indicator-config`。工作台切换不重新加载登录态、医院 ID 或聊天会话。

## 4. 五个业务页面

### 4.1 AI 指标自助配置工作台

复用现有指标设计稿闭环，并补齐需求书要求：

- 自然语言生成指标设计稿；
- 本院设计稿列表、状态筛选和详情；
- 国标模板与本院差异同屏对照；
- 结构化编辑分子、分母、筛选、排除规则和统计周期；
- 元数据候选、字段映射确认；
- SQL 计划预览、DBHub 试运行；
- 提交审批、管理员审批或驳回；
- 历史版本比较和追加式恢复；
- 导出完整指标文档；
- 查看设计稿专用 Trace。

第一阶段先迁移现有弹窗功能，后续再增加结构化编辑、国标对照、恢复和导出，不允许只把原弹窗原样放大。

### 4.2 智能异常排查中心

新增正式诊断页面：

- 选择医院、指标和统计周期；
- 一键执行三层诊断；
- 展示结构、口径、数据三层状态；
- 展示纯国标与本院生效口径的双执行结果；
- 显示根因分类、问题详情、整改建议和修复 SQL 摘要；
- 保存并查询历史诊断报告；
- 跟踪整改状态；
- 重新运行原统计周期并关联验证结果；
- 导出不含患者明细的诊断报告；
- 查看诊断 Trace。

当前只有 `POST /api/diagnose/run`，本页面实施时需要补报告列表、详情、状态更新、导出和重跑关联接口。

### 4.3 口径规则管理后台

将审批、历史版本和医院知识包相关能力整合为规则后台：

- 四个国标指标规则列表和详情；
- 国标记录只读展示，不提供医院端修改或删除；
- 按医院查看当前生效差异；
- 国标与本院差异逐字段对比；
- 待审批变更处理；
- 冲突和逻辑风险提示；
- 医院口径不可变版本历史；
- 追加式恢复历史版本；
- 医院知识包导出；
- 公司端回收、候选审核和版本发布。

页面根据身份显示不同操作：医院人员可提交变更；医院管理员可审批本院变更；公司人员可处理回收候选和公司版本，不能直接修改医院生效口径。

### 4.4 元数据与结构监控看板

将 MCP、元数据同步和系统自检中的结构能力整合：

- DBHub 数据源和工具状态；
- 手工同步元数据；
- 表、字段、类型、主键、外键和可空信息；
- 元数据同步批次和结构变更记录；
- 字段新增、删除、修改统计；
- 指标依赖表和字段清单；
- 字段缺失及受影响指标预警；
- 结构异常定位建议；
- 元数据同步 Trace；
- 失败任务进入恢复中心。

当前已有数据源查看和同步接口，需要增加结构变更列表、指标依赖查询和结构异常汇总接口。

### 4.5 指标运算监控面板

复用已完成的指标监控工作台：

- 日/月运行计划；
- 新建、编辑、启停；
- 手工或指定周期重算；
- 运行结果和 DBHub 耗时；
- 环比、同比及基线状态；
- 波动预警；
- 确认、关闭或重新诊断预警；
- 查看本次运行 Trace；
- 失败后前往恢复中心。

本页面是工作台外壳的首个迁移页面，用于验证导航、管理员回跳、医院切换、响应式布局和 AI 助手共存。

## 5. AI 助手

聊天保留现有 SSE 流式输出、会话记忆、快捷问题和 Trace，但从主页面迁为辅助抽屉：

- 桌面端默认收起，打开后宽度约 `380px`；
- 移动端全屏覆盖，关闭后返回原业务页面；
- 页面切换不清空对话；
- 当前页面可向助手提供非敏感业务上下文，例如当前指标编码、设计稿 ID 或诊断报告 ID；
- 助手不能绕过页面权限执行审批、发布、恢复或预警关闭。

## 6. 权限模型

| 能力 | 医院人员 | 医院管理员 | 公司人员 |
|---|---|---|---|
| 查看本院指标和运行结果 | 是 | 是 | 否 |
| 创建设计稿、提交口径变更 | 是 | 是 | 否 |
| 审批本院口径和设计稿 | 否 | 是 | 否 |
| 元数据同步、运行计划和恢复中心 | 否 | 是 | 否 |
| 查看公司候选和发布公司版本 | 否 | 否 | 是 |
| 修改国标原始记录 | 否 | 否 | 否，必须走公司版本发布流程 |

现有前端管理员密码暂作为医院管理员能力门禁。本批只重组权限入口，不把前端隐藏当作后端授权；所有写接口仍必须在后端校验。

## 7. 前端模块边界

新增：

```text
web/workbench.css
web/workbench.js
web/pages/indicator-config.js
web/pages/diagnosis.js
web/pages/caliber-rules.js
web/pages/metadata.js
web/pages/monitoring-page.js
```

第一批只创建实际使用的文件，不预建空白页面脚本。公共工作台负责：

- Hash 路由；
- 页面导航；
- 当前医院和身份状态；
- 管理员回跳；
- 全局加载和错误提示；
- AI 助手抽屉；
- 页面挂载和卸载。

页面模块负责各自 API、状态、渲染和业务动作。现有 `web/monitoring.js` 先支持弹窗和页面两种容器，迁移稳定后再移除弹窗适配。

## 8. 错误与恢复

- 登录失效：重新登录后返回原页面和原操作上下文；
- 医院切换：清空页面局部选择，按新医院重新加载；
- API 失败：页面保留筛选和已填表单，显示中文原因与重试操作；
- DBHub 异常：指标运行、元数据同步和诊断页面显示对应服务状态，并链接系统自检；
- 长任务：按钮禁用防重复提交，页面显示当前阶段；
- 崩溃恢复：可重试任务继续进入恢复中心，页面不自行重复执行写操作。

## 9. 分批落地

### 第一批：工作台外壳与监控正式页面

- 登录后进入五页面工作台；
- 完成侧边导航、Hash 路由和移动端页面选择器；
- 指标运算监控从弹窗迁为正式页面；
- 聊天迁为可收起 AI 助手；
- 系统自检和恢复中心保留全局入口；
- 其余页面在完成前不显示不可操作的假按钮，可显示明确的建设状态和现有入口，但第一批验收必须以监控页完整可用为准。

### 第二批：AI 指标自助配置工作台

迁移现有设计稿闭环，并补国标对照、结构化编辑、版本恢复、文档导出和专用 Trace。

### 第三批：智能异常排查中心

补报告仓储接口、报告列表/详情、整改状态、导出和重跑验证。

### 第四批：口径规则管理后台

补规则总览、医院差异、冲突提示、版本管理和公司回收发布整合。

### 第五批：元数据与结构监控看板

补结构变更、指标依赖和异常汇总接口，并迁移 MCP 与自检能力。

### 第六批收尾：权限、兼容入口和完整验收

移除重复顶部入口，完成身份矩阵、桌面/移动端视觉验收、可访问性检查、文档和全量回归。

## 10. 验收标准

- 五个需求页面均有独立 URL Hash 和完整业务状态；
- 普通用户不依赖 PowerShell 完成指标配置和异常排查；
- 监控、设计稿、诊断、口径和元数据页面均支持当前医院隔离；
- AI 助手可在任意页面打开，页面切换和关闭助手不丢会话；
- 管理员登录后返回触发操作的原页面；
- 桌面和约 `390x844` 移动视口无文字重叠、横向页面溢出或不可达操作；
- 页面与 Trace 不展示患者明细和含值 SQL；
- 所有新增接口、页面状态和关键交互有自动化测试；
- 全量测试通过，README 更新为五页面日常使用说明。
