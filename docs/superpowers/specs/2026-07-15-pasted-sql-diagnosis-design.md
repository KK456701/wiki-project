# 粘贴 SQL 与执行证据异常诊断设计

## 1. 目标

首版允许医院人员直接在现有 AI 对话框中粘贴 SQL、参数、聚合执行结果和问题描述。系统识别当前会话指标，在医院只读业务库中安全试运行用户 SQL，并与国标口径、本院生效口径和实时数据库元数据进行对比，最终由本地模型组织成医生和实施人员能够理解的诊断说明。

本功能重点验证本地 `qwen3:4b-instruct` 是否能够理解用户问题、SQL 关键语义和结构化检查证据，并生成清晰、克制、可追溯的回答。模型不直接决定 SQL 是否安全，也不独立判断表字段是否存在。

## 2. 首版范围

### 2.1 支持

- 在现有对话输入框中直接粘贴文本，不新增文件上传入口。
- 文本可同时包含问题描述、SQL Server SQL、`DECLARE` 参数和聚合结果。
- 当前会话已有指标时，自动关联该指标；无法确定指标时要求用户补充指标名称。
- 对允许的用户 SQL 自动执行只读试运行。
- 比较用户 SQL、纯国标口径 SQL、本院生效口径 SQL 的结构、条件和结果。
- 在结构和口径没有阻断问题时，执行与当前指标有关的数据质量检查。
- 保存诊断报告、检查证据、模型输出状态和执行链路，不保存患者明细到模型提示词。

### 2.2 暂不支持

- `.sql`、`.txt`、`.csv`、`.xlsx`、PDF 和图片上传。
- 任意未知指标 SQL 的全自动业务语义推断。
- 自动修改或发布医院口径。
- 把患者原始明细发送到公司服务或外部模型。
- 自动执行修复 SQL。

## 3. 设计原则

1. **证据先于结论**：程序先完成结构检查、SQL安全检查、实际试运行和口径差异计算，模型只能基于证据组织语言。
2. **能执行不等于正确**：SQL执行成功只说明语法和数据库对象可用，仍需检查统计范围、时间起点、边界、排除条件、去重和事件选择。
3. **三层检查是内部流程**：前端回答不机械展示“第一层、第二层、第三层”，而是先给结论，再解释原因、影响和处理建议。
4. **医院数据不出院**：真实 SQL 和查询在医院本地执行；模型提示词只包含字段名、规则差异、聚合值和脱敏示例。
5. **诊断不自动改口径**：发现差异后给出建议 SQL，必须经过现有审批发布流程才能成为本院生效口径。

## 4. 输入证据模型

新增类型化的 `PastedDiagnosisEvidence`，代替在诊断链路中继续传递无约束字典：

| 字段 | 说明 |
|---|---|
| `raw_text` | 用户本次粘贴的原始文本，仅用于本地审计和解析 |
| `question` | 用户想排查的问题 |
| `rule_id` | 当前会话确认的指标编码 |
| `sql_text` | 提取到的 SQL 原文 |
| `declared_params` | 从 `DECLARE` 或文本中提取的参数 |
| `claimed_result` | 用户提供的分子、分母、比例等结果 |
| `stat_period` | 识别到的统计开始和结束时间 |
| `parse_warnings` | 无法确认的内容和需要补充的信息 |

本地模型输出只接受固定 JSON 结构。程序会校验字段类型、参数名和指标编码；模型输出无效时使用确定性解析结果继续诊断，并在链路中标记“模型理解结果未采用”。

## 5. 只读 SQL 安全归一化

用户粘贴的 SQL 经过以下处理后才能试运行：

1. 仅允许当前配置的业务数据库，`USE` 指向其他数据库时直接拒绝。
2. 允许若干标量 `DECLARE`，提取参数后从执行语句中移除。
3. 最终只允许一条 `SELECT` 或 `WITH ... SELECT` 查询。
4. 拒绝 `INSERT`、`UPDATE`、`DELETE`、`MERGE`、`EXEC`、动态 SQL、临时表、建表和结构变更。
5. SQL 中引用的数据库和架构必须属于当前业务数据源。
6. 参数使用现有安全字面量绑定，不接受用户拼接的动态对象名。
7. 使用现有只读 DBHub 数据源、查询超时和审计日志。
8. 聚合诊断查询默认只返回聚合结果；需要差异样本时由系统生成受控查询，并限制返回条数。

如果安全归一化失败，系统仍可做静态结构和口径分析，但不会执行用户 SQL，并明确说明未执行原因。

## 6. 诊断流水线

### 6.1 证据识别

- 确定性代码提取代码块、`USE`、`DECLARE`、表名、字段名和聚合别名。
- 本地模型提取用户意图、声称结果和 SQL 的业务含义。
- 两者不一致时以数据库元数据和确定性解析为准，把冲突写入诊断证据。

### 6.2 系统结构检查

- 汇总用户 SQL、国标 SQL 和本院 SQL 使用的表字段。
- 与最新同步元数据比对存在性、类型、可空性和废弃状态。
- “编码字段使用数值类型”不能直接判错；只有运算或比较与实际类型不兼容时才提示。
- 输出内部结构化结果：阻断问题、注意项、缺失对象和适配建议。

### 6.3 口径规则检查

- 独立执行纯国标口径、本院生效口径和用户 SQL。
- 对比统计时间字段、计算起点、时间边界、医院范围、排除条件、去重键、关联方式、事件选择、空值处理和分母保护。
- 识别条件互斥、时间范围重叠或缺口、分母为零、重复计数和条件冗余。
- 对最终值、分子、分母和样本数分别计算差值，不能只比较百分比。
- SQL 已使用 `NULLIF` 或 `CASE` 时，不得误报缺少分母保护。

### 6.4 原始数据检查

- 仅在结构检查没有阻断问题后执行。
- 检查当前指标依赖字段的空值、时间倒置、重复业务键、无效编码和异常事件顺序。
- 优先生成差异原因汇总，例如“统计时间字段不同导致减少59人”“正好48小时边界记录2人”。
- 患者标识只在受控明细页面显示脱敏值，不进入模型提示词和普通诊断回答。

## 7. 面向用户的回答结构

普通回答不显示内部三层名称，采用以下顺序：

1. **先说结论**：结构故障、口径差异、数据质量问题，或证据不足。
2. **为什么不一致**：按影响程度列出2至5项关键原因。
3. **影响了多少**：展示用户 SQL、国标、本院三方的分子、分母、比例及差值。
4. **建议怎么处理**：说明需要业务确认的选择和可执行的修正建议。
5. **查看技术依据**：折叠展示表字段、条件级差异、执行状态和建议 SQL。

示例场景应输出：

> 两段 SQL 都能执行，结果不同主要由统计口径不同造成，不是数据库连接故障。系统当前按入院时间统计，用户 SQL 按首次入区时间统计，因此分母不是同一批患者；此外，两段 SQL 的48小时边界、ICU范围和转科事件选择也不同。

## 8. 当前样例的验收要求

使用用户提供的“患者入院48小时内转科比例”SQL 时，系统必须识别并解释：

- `ADMITTED_AT` 与 `FIRST_ADMITTED_TO_WARD_AT` 的统计范围和计时起点差异。
- `BETWEEN 0 AND 2880` 包含第2880分钟，而 `< DATEADD(HOUR, 48, ...)` 不包含正好48小时。
- 系统配置的ICU组织ID与用户通过 `ORGANIZATION.ORG_NO` 动态取得ICU范围不同。
- 系统选择最早一次有效转科与用户使用 `EXISTS` 判断任意符合事件的差异。
- `<>` 与 `ISNULL(..., 0) <>` 对空值记录的处理差异。
- 用户 SQL 已包含 `NULLIF` 分母保护，不再产生相关误报。
- 数值型ID字段不再被笼统标记为“code类型不匹配”。

用户回答必须先说明“口径差异”，再展示差异证据；内部执行失败或模型JSON解析失败只放在技术依据中，不得用其覆盖已经确认的业务结论。

## 9. 执行链路与审计

诊断链路新增以下节点：

1. `evidence_extract`：提取用户粘贴证据。
2. `user_sql_guard`：安全归一化与只读校验。
3. `user_sql_trial`：自动试运行用户 SQL。
4. `structure_compare`：实时元数据检查。
5. `caliber_semantic_compare`：三方口径条件对比。
6. `data_quality_profile`：按指标执行数据质量检查。
7. `diagnosis_compose`：本地模型组织用户回答。

链路摘要只显示每个节点的状态和一句结论；SQL、参数、模型结构化输出、检查明细和错误码仅在节点详情中展示。诊断报告继续写入现有诊断报告存储，并记录当前规则版本、元数据批次、模型名称和提示词版本。

## 10. 测试策略

- 单元测试：文本证据提取、`DECLARE`解析、跨库拒绝、写入语句拒绝、条件差异识别。
- Agent测试：模型合法JSON、模型无效JSON回退、模型不得改变确定性结论。
- 集成测试：用户 SQL、国标 SQL、本院 SQL 三方只读执行与结果对比。
- 回归测试：原有“诊断当前指标”入口不粘贴 SQL 时保持兼容。
- 样例验收：本设计第8节列出的五类口径差异全部被识别，且不再产生两个已知误报。

## 11. 后续阶段

后续可增加 SQL、文本、CSV、Excel 文件上传，支持更大规模执行明细和差异患者导出。文件解析与当前 `PastedDiagnosisEvidence` 使用同一内部契约，不改变诊断流水线和用户回答结构。
