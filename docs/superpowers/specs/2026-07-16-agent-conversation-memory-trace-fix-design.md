# Agent 会话记忆、Trace 与回答稳定性修复设计

> 日期：2026-07-16  
> 状态：已确认方案 A，待书面规格复核  
> 范围：工具调用型 Agent；旧聊天流程保持兼容

## 1. 背景与问题证据

工具调用型 Agent 已能完成“搜索指标 → 读取本院生效规则 → 回答”，但真实双轮测试暴露以下问题：

1. 模型返回 LaTeX 公式，现有 Markdown 渲染器不支持数学语法，页面直接显示 `$$`、`\text{}`、`\frac`。
2. 新 Agent Trace 使用 `workflow_id=agent_runtime`，项目没有对应 manifest，读取 Trace 时抛出文件不存在并返回 HTTP 500。
3. 新 Agent 前端仍读取旧的匿名 Trace API，没有使用已实现的医院登录态 Agent Run API。
4. 每个新请求创建全新的 `AgentRunState`，`session_id` 只用于 Trace，没有加载结构化状态或历史对话。
5. 第二轮“今年 6 月到现在的这个指标”因此丢失“急会诊及时到位率”，真实 Trace 错误搜索了无关指标并进入重复调用保护。
6. 重复调用在 Gateway 提前返回，未产生完整的 `tool_result` 事件，导致前端和 Trace 不能解释停止原因。
7. 现有回答约束只验证证据类型是否存在，不能阻止模型在没有比较结果时推断“显著升高/降低”。

## 2. 目标

- 工具调用型 Agent 在同一医院、账号和会话内保存完整原文，并在提示预算内使用最近 8 轮用户/助手对话。
- 结构化状态优先于历史原文，至少保存当前指标、统计周期、临时口径和安全对象引用。
- 第二轮“这个指标”能够恢复上一轮指标，并重新读取最新生效规则后继续处理。
- 重复工具调用不重复执行；模型能看到明确的复用/重复结果，持续循环时才停止。
- Agent Trace 可以从前端打开，并通过医院登录态、医院范围和权限校验。
- 页面不再显示未渲染的 LaTeX 控制符。
- 没有确定性比较证据时，回答不得声称指标结果会显著升高或降低。

## 3. 非目标

- 不保存或展示思维链。
- 不保存患者明细、数据库连接、Token、密码、完整 SQL 文本或原始敏感工具结果。
- 不增加自动提交、审批、发布或回退能力。
- 不删除旧聊天流程，不修改其现有 8 轮会话行为。
- 不新增通用向量记忆、跨会话用户画像或长期知识学习。

## 4. 方案选择

### 4.1 采用：复用 `ConversationMemory`

复用现有 SQLite + JSONL 会话存储和 `ConversationContext`：

- 原始用户/助手消息继续写入 `messages`；
- 当前指标、统计周期和临时口径写入 `session_contexts`；
- SQL ID、运行 ID、诊断 ID、草稿 ID 只作为受限元数据保存；
- 提示词通过现有 `build_prompt_context` 截取最近 8 轮并执行字符预算压缩。

该方案与旧流程共用成熟的数据契约，但使用独立的内部会话键，避免同一浏览器会话 ID 被不同医院或账号复用。

### 4.2 不采用：持久化整个 `AgentRunState`

完整状态包含工具结果、证据和可能已过期的 SQL 对象引用，容易扩大敏感数据范围并造成过期状态误用。

### 4.3 不采用：浏览器上传历史

客户端历史可被篡改，刷新或更换设备后不可靠，也不能作为医院业务状态的权威来源。

## 5. 会话身份与数据边界

前端继续使用公开 `session_id`。服务端生成内部存储键：

```text
agent:{hospital_id}:{user_id}:{session_id}
```

只有经过医院 Bearer Token 认证后才能加载该键。相同公开 `session_id` 在不同医院或账号下不会共享记忆。

存储内容：

- 用户问题和最终助手回答完整原文；
- 当前规则 ID、规则名称、医院 ID；
- 统计开始与结束时间；
- 当前会话临时口径；
- 仍需由领域服务再次校验的 `sql_id`、`run_id`、`diagnosis_id`、`draft_id`。

禁止存储到会话记忆：

- 工具原始参数和完整原始结果；
- 患者明细、样例值和导出内容；
- SQL 正文、连接串、Token、密码和内部提示；
- 模型推理过程。

## 6. 每轮数据流

### 6.1 请求开始

1. API 根据登录主体和公开 `session_id` 生成内部存储键。
2. `ConversationMemory.ensure_session` 确保会话存在并绑定医院。
3. 加载 `ConversationContext` 和最近 16 条用户/助手消息，即最多 8 轮。
4. 使用 `build_prompt_context` 生成结构化摘要和有界历史；结构化摘要明确标记为权威来源。
5. 从结构化状态恢复 `current_rule_id`，从安全元数据恢复仍可能有效的对象 ID。
6. 将上一轮已验证的规则身份恢复为“会话记忆证据”，仅用于开放 `get_effective_rule`；定义、公式和当前版本仍必须重新调用该工具取得。
7. 将当前日期写入系统上下文，帮助模型解析“今年 6 月到现在”。

### 6.2 Agent 循环

- 如果有 `current_rule_id`，模型可以直接调用 `get_effective_rule`，不必再次模糊搜索。
- 如果用户询问实际数值，模型必须准备受控 SQL 并进行只读试运行；只有规则证据不能支持实际数值。
- 同一工具和相同参数首次重复时，Gateway 返回已有安全结果或明确的“结果已存在”事件，不再次执行工具。
- 再次持续重复时停止循环，返回可理解的澄清或失败信息，并在 Trace 中记录重复指纹的次数，不显示原始敏感参数。

### 6.3 请求结束

1. 保存最终助手回答。
2. 从经过校验的工具结果提取当前规则 ID、规则名称和统计周期。
3. 保存结构化上下文；版本冲突时不覆盖新版本，并记录固定问题码。
4. 保存安全对象引用；实际使用时仍由对象存储校验医院、用户、会话、TTL 和状态。
5. 不把本轮 `tool_call_counts`、原始工具结果或模型消息直接持久化。

## 7. 公式与事实约束

系统提示新增明确约束：

- 使用普通中文 Markdown；
- 公式格式为 `指标率 = 分子 ÷ 分母 × 100%`；
- 禁止输出 `$$`、`\(`、`\)`、`\[`、`\]`、`\text`、`\frac`；
- 没有确定性对比或试运行证据时，只能说明“口径不同，结果不可直接比较”，不得判断显著升高或降低。

后端在最终回答进入事实校验前执行最小规范化：

- 删除独立数学分隔符；
- 将简单 `\text{内容}` 转为“内容”；
- 将简单 `\frac{分子}{分母}` 转为“分子 ÷ 分母”；
- 将 `\times`、`\%` 转为 `×`、`%`。

前端 Markdown 渲染器把独立 `---` 渲染为分隔线，避免原样显示。

## 8. Trace 修复

新增 `app/workflows/agent_runtime.yaml`，登记三个稳定节点：

- `agent_model`：模型选择下一步；
- `agent_tool_call`：请求执行受控工具；
- `agent_tool_result`：工具返回安全摘要和证据来源。

步骤号、工具名和风险等级放在节点配置中，不再把步骤号或工具名拼进 manifest 节点 ID。

工具调用型回答的“查看链路”改为：

```text
GET /api/agent/runs/{trace_id}
Authorization: Bearer <hospital token>
```

旧流程回答继续使用 `/api/traces/{trace_id}`，避免改变现有行为。Agent Run API 保持医院范围校验，其他医院的 Trace 返回 403。

## 9. 错误处理

- 会话记忆读取失败：新 Agent 明确返回“会话状态暂不可用”，不把“这个指标”当作新问题猜测。
- 会话记忆保存失败：本轮已经取得的回答可以返回，但 Trace 标记 `AGENT_MEMORY_SAVE_FAILED`，前端提示后续追问可能无法继承。
- 历史对象过期：领域对象存储返回固定过期问题码，模型重新准备对象，不绕过 TTL。
- Trace manifest 缺失：Trace 读取器降级为未登记节点，而不是返回 HTTP 500。
- 重复调用达到停止门槛：记录完整事件后结束，不显示笼统且无法排查的空链路。

## 10. 测试设计

### 10.1 会话记忆

- 第一轮读取 `MQSI2025_005`，第二轮“这个指标”恢复规则 ID。
- 最近 9 轮时只向模型提供最近 8 轮，存储中仍保留全部原文。
- 相同公开会话 ID、不同账号或医院不能共享历史。
- 结构化当前指标优先于历史中较旧的指标名称。
- 工具原始结果、SQL 文本和敏感字段不进入消息元数据。
- SQL ID 过期后不能因记忆恢复而绕过 TTL。

### 10.2 Agent 循环

- 有记忆规则身份时首个可见工具包含 `get_effective_rule`。
- 第二轮不再强制搜索无关指标。
- 首次相同调用不重复执行，并产生可见的复用事件。
- 持续重复才停止，Trace 含固定问题码和调用次数。

### 10.3 回答格式与事实

- LaTeX 示例被规范成普通中文公式。
- 普通 Markdown 不受影响。
- 没有比较证据时，系统提示禁止推断升降方向。
- 前端 `---` 渲染为 `<hr>`，不显示三个减号。

### 10.4 Trace

- `agent_runtime` Trace 返回 200 且节点有中文标题。
- 新 Agent 前端请求登录态 Agent Run API并携带 Bearer Token。
- 其他医院不能读取 Trace。
- 旧流程 Trace API 保持原行为。

### 10.5 回归与真实验收

- Agent 相关单元与 API 测试通过。
- 全量 `python -m pytest -q` 通过。
- 浏览器真实验证：
  1. 输入“急会诊及时到位率怎么算？”；
  2. 输入“我想看今年 6 月到现在的这个指标是多少怎么算”；
  3. 第二轮使用同一指标，调用规则、SQL 准备和只读试运行；
  4. 回答无 LaTeX 控制符；
  5. 两轮“查看链路”均可打开。

## 11. 验收标准

1. 同一会话至少保留并使用最近 8 轮用户/助手对话。
2. “这个指标”不会重新猜测无关指标。
3. 统计周期进入结构化状态并用于受控 SQL。
4. 重复工具不会重复访问领域服务或数据库。
5. Agent Trace 通过登录态接口成功加载，并执行医院隔离。
6. 最终回答不出现未渲染 LaTeX 控制符。
7. 无对比证据时不产生升降方向断言。
8. 不扩大患者数据、凭据、SQL 正文和内部推理的存储与展示范围。
