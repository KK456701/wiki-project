# 指标监控工作台设计

## 1. 目标

将第五批已经完成的指标运行计划、手工重算、运行结果、预警和执行链路能力接入现有前端，使医院管理员不依赖 PowerShell、接口文档或数据库客户端即可完成日常监控操作。

本次继续采用现有单页应用和管理弹窗，不引入新的前端框架。完整工作台作为顶部一级入口，与系统自检、恢复中心、指标设计稿和审批并列。

## 2. 用户与权限

- 主要用户：医院医务、质控管理员和本地实施人员。
- 当前医院：默认读取页面顶部的医院 ID，所有计划、结果和预警请求都显式传递该 ID。
- 管理权限：打开监控工作台时复用现有管理员登录流程；管理员令牌失效后重新弹出登录窗口，并在登录成功后返回监控工作台。
- 普通业务用户不需要执行命令行。manifest 校验和原始 API 保留给开发、自动化测试和实施排障。

## 3. 界面结构

顶部增加“指标监控”按钮，点击后打开宽屏管理弹窗。弹窗使用三个标签：

1. **运行计划**：左侧为计划列表，右侧为当前计划详情、启停状态、波动阈值和最近一次运行结果。
2. **运行结果**：按当前医院列出运行记录，可按指标编码筛选；展示统计周期、结果、环比、同比、状态、耗时和执行时间。
3. **预警处理**：列出当前医院预警，展示指标、预警类型、变化率、诊断状态和处理状态。

已确认的运行计划布局为：左侧计划列表，右侧计划详情和最近结果。移动端改为单列，计划列表位于详情上方；按钮和状态不得与文本重叠。

## 4. 运行计划交互

### 4.1 新建与编辑

表单字段：

- 指标编码；
- 计划名称；
- 频率：每日或每月；
- 运行时间；
- 每月执行日，仅月计划显示，范围 1 至 28；
- 环比开关和阈值，默认启用、20%；
- 同比开关和阈值，默认启用、30%。

表单提供中文标签、合理默认值和内联错误。保存成功后刷新计划列表并保持当前计划选中；保存失败时保留用户输入。

### 4.2 启停与手工运行

- 启用和停用使用清晰的状态按钮，操作完成后同步刷新计划详情。
- “立即运行”打开轻量确认区，默认使用系统计算的最近完整周期，也允许输入明确日期范围。
- 运行期间按钮显示处理中并禁止重复点击。
- 运行成功后切换到最近结果摘要，并提供“查看执行链路”按钮。
- 运行失败时显示问题详情和“前往恢复中心”，不要求用户复制命令。

## 5. 运行结果与执行链路

结果列表读取 `/api/monitoring/results`，展示：

- 指标编码、统计周期和触发方式；
- 聚合结果；
- 环比和同比变化率；
- `success`、`no_sample`、`failed` 的中文状态；
- DBHub 执行耗时和创建时间。

手工运行响应包含 `trace_id` 时直接调用现有 `showTrace(trace_id)`。执行链路沿用已实现的七个节点：计划读取、运行锁、统计周期、DBHub 运算、波动判断、预警创建和自动诊断。节点详情只展示聚合值、版本、数据源、耗时、预警和报告 ID，不展示 SQL 或患者明细。

历史运行记录目前未持久化 `trace_id`，因此仅对本次手工运行直接提供链路入口；列表中没有 `trace_id` 的记录不显示无效按钮。后续如需从所有历史记录回看链路，应单独扩展结果表和 API。

## 6. 预警处理

预警标签读取 `/api/monitoring/alerts`，提供：

- 按未处理、已确认、已关闭筛选；
- 确认预警；
- 关闭预警；
- 诊断失败时重新诊断；
- 本次运行在当前页面产生 `trace_id` 时查看关联链路。

确认和关闭按钮在请求完成前禁用。接口失败时保留当前列表和筛选条件，并在对应预警内显示处理建议。

## 7. 状态与错误处理

每个标签独立处理以下状态：

- 加载中：显示当前正在读取的内容，不使用空白区域代替。
- 空数据：说明当前没有计划、结果或预警，并给出下一步操作。
- 权限失效：清除无效令牌，重新打开管理员登录；登录后返回监控工作台。
- 调度器不可用：在工作台顶部显示异常状态和系统自检入口，手工操作按后端返回结果处理。
- 网络或后端失败：显示中文错误和可执行的处理建议，不只显示“请求失败”。
- 重复操作：按钮在进行中禁用，完成后统一恢复。

## 8. 前端代码边界

继续使用 `web/index.html`，但监控功能按职责拆分：

- API 请求：统一管理员请求头、错误解析和令牌失效处理；
- 状态管理：当前标签、计划、结果、预警和最近手工运行 Trace；
- 渲染函数：计划列表、计划详情、结果列表、预警列表分别实现；
- 操作函数：保存、启停、运行、确认、关闭和重新诊断分别实现；
- 通用能力：复用 `requireAdminThenOpen()`、`showTrace()` 和现有关闭弹窗逻辑。

监控函数不得修改聊天会话状态，也不得复制执行链路渲染代码。

## 9. Agent 工程原则

在根目录 `agent.md` 增加以下硬性要求：

- **可用性**：用户日常业务操作必须在前端完成；加载、空数据、成功、失败和权限失效都有明确反馈；禁止把命令行当作普通用户主流程。
- **可维护性**：模块职责清楚，复用现有接口和组件，避免大函数、重复请求逻辑和同义状态；功能修改必须同步测试和文档。
- **易上手**：界面使用业务语言、合理默认值和就地提示；关键操作路径短且可恢复；新用户无需理解 MCP、manifest、run_key 等内部术语。

命令行、原始 JSON 和 manifest 仍可用于开发与实施排障，但不能替代产品界面。

## 10. 测试与验收

### 10.1 自动化测试

- 页面包含监控入口、工作台弹窗、三个标签和计划表单；
- 页面调用计划、结果、预警、运行和诊断相关 API；
- 管理员登录成功后能返回监控工作台；
- 计划和结果状态有中文映射；
- 本次运行的 `trace_id` 调用现有执行链路；
- 页面不包含需要普通用户执行的 PowerShell 提示；
- 现有后端和前端测试全部通过。

### 10.2 浏览器验收

1. 医院人员登录，顶部医院 ID 为 `hospital_001`。
2. 点击“指标监控”，管理员未登录时先完成管理员登录并自动进入工作台。
3. 新建 `MQSI2025_005` 月计划，验证默认环比 20%、同比 30%。
4. 编辑、停用和重新启用计划，列表与详情状态同步。
5. 手工运行明确统计区间，看到聚合结果、变化率、预警和真实耗时。
6. 点击“查看执行链路”，看到七个监控节点且无 SQL 和患者明细。
7. 在预警标签确认、重新诊断或关闭预警。
8. 在桌面和移动视口检查无文字溢出、按钮重叠和不可见内容。

验收产生的临时计划、运行结果、预警和诊断报告必须使用专用标识，并在验收结束后删除。
