# 执行链路双层摘要设计

## 1. 目标

将现有执行链路从“所有观测信息平铺”调整为“业务摘要 + 技术详情”双层结构，使医院业务人员能快速了解本次请求做了什么、结果是什么、哪里耗时，同时保留实施和开发人员需要的输入输出、节点配置、字段检查与故障定位信息。

本次不改变工作流执行顺序，不删除 Trace 原始数据，不改变“用户主动展开后查看详情”的既有交互原则。

## 2. 当前问题

1. 正常节点也持续展示必要字段、出错处理、问题码和定位建议，视觉上接近故障页面。
2. `effective_rule_resolve` 的 Trace 实际输出是安全摘要字段，但 manifest 要求嵌套 `effective_rule`，导致成功节点产生契约假告警。
3. 节点描述比本次处理结果更突出，用户无法快速看出“命中了哪个指标、采用哪个口径”。
4. 顶部耗时使用节点耗时相加，和 Trace 真实墙钟耗时不一致，也没有展示最慢阶段占比。
5. 原始 JSON、节点定义和排障信息没有清晰分层。

## 3. 总体结构

### 3.1 第一层：业务摘要

打开执行链路后默认显示：

- 总体状态、真实总耗时、成功节点数；
- 阶段时间轴；
- 每个节点的序号、中文标题、中文状态、耗时；
- 本次业务结果摘要，而不是通用职责描述。

首批五个对话节点摘要示例：

| 节点 | 业务摘要 |
|---|---|
| `memory_load` | 无历史指标上下文 / 已读取上轮指标上下文 |
| `intent_detect` | 识别为指标口径查询 / SQL 生成 / 异常诊断等 |
| `rule_search` | 命中指标编码、命中候选数量 |
| `effective_rule_resolve` | 本院口径或国标口径、版本、MySQL 或 Wiki 来源 |
| `final_response` | 回答已生成 / 回答生成失败 |

### 3.2 第二层：技术详情

点击节点“详情”后显示三个区域：

1. **处理结果**：将关键输出转为可扫描的字段和值。
2. **输入输出**：原始输入、输出和节点配置，默认再次折叠，用户主动点击后查看 JSON。
3. **开发与排障**：负责 Agent、必要输入输出、检查状态、出错处理和问题码。

成功且字段检查正常的节点不展示黄色定位建议。节点失败、回退、警告或字段检查异常时，自动展示缺少字段、错误信息和定位建议。

## 4. 三批落地范围

### 4.1 第一批：数据正确性

- 顶部总耗时使用 Trace 自身的 `duration_ms`，不再用节点耗时之和冒充总耗时。
- 节点耗时之和保留为“已记录阶段耗时”，只用于性能覆盖说明。
- 校准 `core_indicator_chat.yaml` 的观测输入输出，使其与 Trace 实际保存的安全字段一致：
  - `rule_search` 输出采用 `rule_id`、`matched_count`、`context_source`；
  - `effective_rule_resolve` 输出采用 `rule_id`、`effective_level`、`rule_source`、版本和差异字段；
  - `final_response` 输出采用 `answer_preview`、`trace_id`、`final_status`。
- `effective_rule_resolve` 的必要输出改为 `rule_id`、`effective_level`、`rule_source`，消除成功节点假告警。
- 契约异常与执行状态分开判断：执行成功但观测字段异常时显示“字段检查异常”，不能伪装成执行失败。

### 4.2 第二批：节点双层摘要

- 新增稳定的节点业务摘要函数，按节点类型读取安全输出字段。
- 折叠状态只显示序号、标题、业务摘要、中文状态和耗时。
- 节点 ID、类型、工具和数据源移入技术详情。
- 详情按“处理结果、输入输出、开发与排障”分组。
- 原始 JSON 使用二级折叠，不在点击节点详情后立即铺满页面。
- 状态统一映射为中文：成功、失败、已回退、需关注、进行中、未知。
- 失败节点自动展开技术详情；正常节点保持折叠。

### 4.3 第三批：阶段时间轴与性能定位

- 顶部新增五阶段时间轴，按执行顺序展示状态和耗时。
- 每个已计时节点显示占真实总耗时的百分比。
- 最慢节点显示“最慢”标记；单节点超过总耗时 50% 或耗时超过 3 秒时显示性能提示。
- 时间轴在桌面端横向排列，移动端改为纵向列表，不产生横向溢出。
- 未计时节点显示“未计时”，不参与百分比和最慢节点计算。

## 5. 数据与安全边界

- 前端只消费 `/api/traces/{trace_id}` 已返回的安全 Trace 数据。
- 不新增患者明细、数据库密码或绑定后的 SQL。
- 监控链路已有安全字段过滤继续生效。
- 原始 JSON 仍进行 HTML 转义，不能通过 Trace 内容注入页面。

## 6. 状态与异常处理

- Trace 加载中显示明确加载状态。
- 无节点时说明当前链路没有阶段记录。
- Trace 加载失败时显示错误原因和可重试按钮。
- 契约检查异常只在对应节点显示，不污染其他成功节点。
- 总耗时缺失时显示“未计时”；不能回退为不准确的节点耗时之和。

## 7. 测试与验收

### 7.1 自动化测试

- Trace 总览使用 Trace `duration_ms`。
- 生效口径成功节点不再缺少 `effective_rule`。
- 五个核心节点均生成业务摘要。
- 成功节点默认不展示定位建议，异常节点展示。
- 原始 JSON 位于二级折叠区。
- 状态全部使用中文映射。
- 时间轴包含节点耗时、占比和最慢标记。
- 移动端时间轴为单列且无横向溢出。

### 7.2 浏览器验收

使用“急会诊及时到位率怎么算？”产生五节点链路：

1. 顶部显示真实总耗时和 5/5 节点。
2. 默认一屏可看到五阶段摘要，不展开原始 JSON。
3. 意图识别节点显示最慢标记和耗时占比。
4. 生效口径节点显示“本院口径 · MySQL · v1”，且没有缺少 `effective_rule` 的黄色假告警。
5. 展开会话记忆节点后，先看到处理结果；再次展开“原始输入输出”才看到 JSON。
6. 正常节点不显示定位建议；模拟失败节点时能看到问题码、错误和定位建议。
7. 桌面与移动视口均无文字重叠、按钮溢出和页面级横向滚动。
