# 医生友好的指标计算说明设计

## 目标

将“生成 SQL”和“试运行”的回答改为医生可以直接理解的业务说明。普通用户不需要理解业务字段、数据库字段、表名或 SQL，也能知道分母、分子、医院口径和最终比例是怎么得到的。

## 展示原则

1. 默认展示业务解释，技术信息默认折叠。
2. 先讲“统计了谁”，再讲“哪些进入分子”，最后展示计算结果。
3. 分子必须明确说明它继承全部分母条件，并额外满足什么条件。
4. 医院口径必须说明本院值、标准值及它影响分子还是分母。
5. 不根据字段名称猜测业务含义；结构化关系缺失时明确提示需要实施人员完善配置。

## 默认医生视图

### 生成 SQL 后

回答依次展示：

1. **当前指标口径**：指标名称、统计医院、统计时间、本院采用的规则。
2. **分母怎么得到**：使用完整自然语言说明进入统计范围的记录。例如“统计本院在本月发起的全部急会诊，共同构成分母”。
3. **分子怎么得到**：先说明分子来自分母，再说明额外条件。例如“在上述急会诊中，申请后 0 至 20 分钟内到达的记录进入分子”。
4. **本院规则有什么不同**：例如“国家标准为 10 分钟，本院生效规则为 20 分钟；该变化只影响分子，不改变分母”。
5. **计算公式**：使用指标业务名称表达，不显示字段表达式。
6. **查看技术详情**：默认折叠，包含数据库位置、字段映射和 SQL。

### 试运行后

在上述说明基础上增加一段可直接复述的结论：

> 本期共有 576 次急会诊进入统计范围，其中 488 次在本院规定的 20 分钟内到达，另有 88 次未在规定时间内到达。因此，急会诊及时到位率为 488 / 576 × 100% = 84.72%。

同时使用简表展示：分母数量、分子数量、未进入分子的数量和最终结果。

## 技术详情

技术详情供信息科和实施人员排障，默认折叠。原有列名调整为：

| 原名称 | 新名称 |
| --- | --- |
| 业务字段 | 系统统一名称 |
| 医院表字段 | 本院数据库位置 |
| 判断或计算方式 | 系统如何判断 |
| 条件来源 | 规则来源 |

SQL、安全校验结果、数据库表名和字段名只在该区域展示。折叠区域不得影响原有字段血缘和故障定位能力。

后端使用固定的技术详情分段标记输出内容，前端聊天渲染器将该分段转换为原生 `details/summary` 折叠区域。前端不直接信任或执行回答中的 HTML，避免为了折叠功能放开不安全的 HTML 渲染。

## 数据与生成方式

后端继续使用结构化字段血缘生成说明，不由大模型自由编造：

1. 指标计算定义提供分母、分子及派生关系。
2. 本院生效口径提供实际阈值和影响范围。
3. 字段映射提供技术详情中的数据库位置。
4. 试运行只使用聚合结果生成数量说明，不展示患者明细。

自然语言说明与技术详情必须来自同一份结构化数据，避免两套解释不一致。

## 异常与降级

1. 字段关系未结构化时，医生视图显示“当前指标的取数关系尚未配置完整，请联系信息科或实施人员完善后再生成”，不展示推测内容。
2. 分母为零时显示“本期没有符合统计范围的数据，暂不能计算该指标”，不得显示 `0 / 0`。
3. 分子大于分母时显示结果异常，并引导检查本院规则或数据映射。
4. 旧版 SQL 没有返回分子、分母时，提示重新生成 SQL。

## 验收标准

1. 默认可见区域中不出现表名、字段名和 SQL。
2. 医生无需展开技术详情即可说明分母、分子、医院口径和指标结果。
3. 展开技术详情后仍可看到完整字段映射和 SQL。
4. 急会诊示例明确说明 20 分钟为本院口径、10 分钟为标准口径，并说明只影响分子。
5. 试运行明确解释 `488 / 576 × 100% = 84.72%`，同时说明 88 次未及时到位。
6. 自动化测试覆盖生成说明、试运行说明、技术详情折叠标记和异常降级路径。
