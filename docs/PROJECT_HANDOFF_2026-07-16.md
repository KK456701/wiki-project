## 2. 项目目标

项目面向医院医务、质控和实施人员，提供核心制度指标的：

- 规则维护、医院口径适配、审批、版本化和回退。
- 医院数据库结构同步、字段映射确认和只读 SQL 试运行。
- 分子、分母、指标结果和患者级明细核对。
- 用户 SQL 与当前生效 SQL 的差异诊断。
- 指标运行计划、波动预警和自动诊断。
- 医学术语、同义词和医院编码映射维护。
- 公司标准包下发与医院反馈包回收。

项目不是单纯的聊天机器人。更准确的产品定位是：

> 核心制度指标实施与运行平台，AI 作为自然语言理解、设计辅助和诊断解释工具。

## 3. 最新架构判断

### 3.1 已发布指标不应由本地小模型临时写 SQL

当前正式 SQL 生成链路已经基本采用确定性执行：

```text
MySQL 读取已审批指标规则和 SQL 模板
→ 合成本院生效口径
→ 读取本院字段映射和业务编码参数
→ 绑定统计区间、医院和口径参数
→ 生成只读 SQL
→ 元数据预检查和安全校验
→ DBHub 在医院业务库只读试运行
```

因此前端“生成 SQL”的实际含义是：

> 根据已审批指标方案，生成本次统计可执行 SQL。

它不是每次调用 Ollama 从零编写 SQL。

### 3.2 推荐的最终分层

建议后续明确分成三个产品阶段：

#### 公司指标工厂

- 使用公司侧大模型解析制度文件、生成指标设计草稿和候选 SQL。
- 结合 WxP 公司标准表模型、数据字典和允许回收的医院纯元数据做字段候选匹配。
- 在公司测试环境或受控医院测试环境完成试运行。
- 经过业务、产品、研发审核后，发布带版本和签名的指标实施包。
- 公司侧不得收集患者明细；医院元数据能否回收由医院安全制度决定。

#### 医院实施控制台

- 导入公司发布的指标实施包。
- 同步医院 `INFORMATION_SCHEMA`，不读取患者明细。
- 由实施人员确认公司业务概念与医院实际表字段的映射。
- 配置医院组织编码、业务类型、ICU 范围和时间阈值等参数。
- 在医院只读测试库试运行，查看分子、分母和差异明细。
- 审批后发布为本院生效版本。

#### 医院生产运行

- 只使用已审批 SQL 模板、字段映射、口径参数和统计区间。
- 不允许模型在生产请求中临时生成任意 SQL。
- 本地小模型只负责可选的问答、自然语言解释和诊断摘要。
- 即使 Ollama 不可用，指标计算、试运行、监控和审批仍应可用。

### 3.3 尚未完成的产品重心调整

当前前端仍以“AI 指标助手”为主入口。下一阶段建议改成：

- 指标实施控制台和指标运行工作台作为主要业务入口。
- AI 助手作为右侧辅助入口或上下文助手。
- 固定指标的选择、映射、试运行、发布和监控全部可以不用自然语言完成。
- 不懂 SQL 的实施人员通过表单、字段候选、业务解释和差异明细完成工作。

这项产品重心调整目前只是明确建议，尚未完成整体前端和导航重构。

## 4. 当前 LLM 实际用途

本地模型默认通过 Ollama 调用 `qwen3:4B-instruct`。当前用到 LLM 的位置包括：

| 场景 | 当前作用 | 失败后的处理 |
|---|---|---|
| 意图理解 | 识别查询、生成 SQL、试运行、反馈、诊断等意图，提取指标名、筛选条件和上下文更新 | 规则识别兜底 |
| 普通指标问答 | 把已检索的规则组织成自然语言回答 | 知识库模板兜底，公式事实校验 |
| 会话追问 | 辅助理解“这个指标”“按入区算”等表达 | 结构化会话状态是权威来源 |
| 指标设计稿 | 把自然语言指标需求解析为分子、分母、统计周期和结构化计划 | 解析失败则阻止进入后续实施 |
| 粘贴 SQL 诊断 | 辅助提取用户说明、参数和 SQL 证据 | 确定性提取兜底 |
| 诊断报告表达 | 把结构校验、口径比较和数据校验结果组织成可读结论 | 固定中文报告模板兜底 |

明确不依赖 LLM 的关键环节：

- 已审批指标 SQL 模板读取和参数渲染。
- 医院字段映射读取。
- 元数据字段存在性与类型检查。
- SQL 只读安全校验。
- DBHub 试运行。
- 分子、分母、患者明细和差异明细计算。
- 审批、版本化、回退、监控和审计。

相关代码入口：

- `app/agents/human_interaction.py`：意图识别、问答和规则兜底。
- `app/indicators/parser.py`：自然语言指标设计稿解析。
- `app/diagnose/evidence.py`：诊断证据提取。
- `app/diagnose/narrator.py`：诊断报告组织和固定模板兜底。
- `app/sqlgen/agent.py`：确定性 SQL 生成、校验和试运行。
- `app/agent/graph.py`：主聊天链路、会话状态和 SSE 流式适配。

## 5. 数据与知识架构

### 5.1 MySQL 是权威事实层

`wiki_agent_runtime` 保存：

- 国标指标规则和结构化计算定义。
- 医院定制口径、审批状态、生效时间和版本。
- SQL 模板、字段映射和参数。
- 元数据快照、结构变更和受影响指标。
- 监控计划、运行结果、预警、诊断和审计。

国标原始记录不被医院直接修改。查询时以国标为基础，合入本院已审批且生效的差异项，形成“本院生效口径”。

### 5.2 Wiki 是导入来源和只读兜底

`core-rules-wiki/` 保存可审阅的制度文档、YAML 规则、SQL 规格、字段映射和术语来源。

- MySQL 正常时，以 MySQL 生效版本为准。
- MySQL 异常或规则尚未迁移时，Wiki 只读兜底。
- 审批写入、版本回退和生产状态不写回 Wiki。

### 5.3 医院业务数据不出院

- 医院患者业务数据保存在医院 SQL Server。
- FastAPI 通过 DBHub MCP 使用只读账号查询。
- 元数据同步主要读取 `INFORMATION_SCHEMA`。
- 患者明细预览必须脱敏。
- 完整 Excel 仅允许授权人员导出，服务器临时文件默认 24 小时清理。

### 5.4 WxP 的边界

`tools/wxp-mcp` 是公司侧标准表模型工具，用于查询公司标准表、字段、索引和血缘候选。

- 不进入医院生产链路。
- 不替代医院 DBHub 和真实 `INFORMATION_SCHEMA`。
- WxP 候选必须与医院实际数据库、数据字典和指标口径逐项确认。
- Session、内部仓库地址、查询结果和源码 vendor 均不得提交 Git。

### 7.5 Agent 工具调用运行时与阶段 7 灰度闭环

工具调用型 Agent 的阶段 1—7 已完成，旧聊天没有删除：

- 独立 Runtime、Gateway、风险策略、Ollama 工具调用适配器和最多 8 步循环。
- 指标检索、生效规则、实施状态、SQL 准备/只读试运行、诊断、设计草稿和口径差异预览等复合工具。
- `/api/agent/chat/stream`、`/api/agent/capabilities`、Agent Trace 和安全 SSE 事件。
- 新 Agent 每轮恢复结构化状态，并将最近 8 轮完整对话放入模型提示；历史原文只辅助理解，当前生效规则仍由工具重新读取。
- Agent 会话存储键按 `hospital_id + user_id + session_id` 隔离；只持久化当前指标和受控对象引用，不保存工具结果缓存、SQL 正文、患者明细或敏感参数。
- 同一轮内第二次同参工具调用复用第一次安全 `ToolResult`，第三次才停止循环；复用缓存不跨请求、不写入会话记忆。
- 新 Agent 的“查看链路”使用登录态 `/api/agent/runs/{trace_id}`，后端按医院校验；旧回答继续使用原 `/api/traces/{trace_id}`。
- 64 个固定业务案例，覆盖 16 类问法；关键与安全案例重复三次，任一安全违规整批失败。
- `shadow` 仅后台调用三个低成本只读工具，不改变旧回答、不重复 SQL 或诊断。
- 内部实施账号灰度入口、模式标识和只展示安全业务事件的证据轨迹。

默认配置仍是 `agent_enabled: false`、`agent_mode: legacy`。上线不得直接全量打开，必须按“固定评测 → Shadow → 指定医院实施账号 → 扩大灰度”推进。具体命令、阈值、监控和紧急回退见 `docs/operations/agent-tool-calling-rollout.md`。

## 8. 会话上下文现状

当前采用“结构化状态 + 最近原始消息”的方案：

- 结构化状态保存当前指标、医院、统计区间和临时口径，是 SQL 生成和试运行的权威输入。
- 最近原始消息只帮助本地模型理解自然语言追问。
- 当前轮明确统计区间优先于历史区间。
- “生成 SQL”“试运行”等短指令继承最近一次有效指标和统计区间。
- 本地模型默认上下文配置为 16K，而不是 Ollama 默认的 4096。
- 新 Agent 的内部存储会话 ID 同时包含医院、用户和公开会话 ID，避免相同 `session_id` 在用户或医院之间串话。
- 保存全部消息用于审计和恢复，但模型提示最多装载最近 8 轮，并继续受提示 Token 预算限制。

仍需继续关注：所有业务参数修改都必须进入结构化状态，不能只存在模型回答文本中。

## 10. 建议验证流程

### 10.1 急会诊指标

1. 输入“急会诊及时到位率怎么算？”。
2. 在同一会话输入“我想看今年6月到现在的这个指标是多少怎么算”。
3. 核对第二轮仍命中 `MQSI2025_005`，没有检索无关指标，并完成生效规则、实施状态、SQL 安全准备和只读试运行。
4. 核对公式显示为普通文本 `指标率 = 分子 ÷ 分母 × 100%`，不出现 `$$`、`\frac`、`\text` 等控制符。
5. 分别点击两轮回答的“查看链路”，确认登录态 Trace 能正常加载。
6. 查看分母、分子、未达标明细和 Excel 导出。

### 10.2 入院 48 小时转科指标

1. 输入“患者入院 48 小时内转科的比例怎么算？”。
2. 修改统计区间或临时口径。
3. 输入“生成 SQL”，检查统计区间是否继承。
4. 打开技术详情，复制 Navicat 可执行 SQL 验证。
5. 输入“试运行”，核对分子、分母和明细。

### 10.3 用户 SQL 差异诊断

1. 在问题中明确指标名称。
2. 粘贴只读 SQL 和可选的聚合结果。
3. 系统应展示用户 SQL 与当前生效 SQL 的执行状态、分子、分母和指标值。
4. 展开技术依据时不应看到原始 `<details>` 文本。
5. 支持逐条对账时，应出现“查看差异明细”按钮。
6. 点击后检查五类差异、分页、数据源和脱敏标识。

注意：历史回答不会因代码升级自动补按钮，必须重新发起一次诊断。

## 13. 下一阶段推荐顺序

Agent 工具调用架构阶段 1—7 已完成。下一阶段优先做真实模型评测和受控灰度，不再新增固定意图分支来掩盖模型工具选择失败：

1. 用目标 Ollama 模型生成固定评测集真实观测，先达到所有能力与安全门槛；
2. 在 `shadow` 下观察完整业务周期，确认只读工具选择、证据一致性和 DBHub 压力；
3. 仅对指定医院、具备实施权限的内部账号启用 `tool_calling`；
4. 4B 模型仍不达标时，先优化工具说明和每轮暴露数量，再对比 8B 或更强的受控模型；
5. 旧流程继续作为独立回退路径，暂不删除。

### 第一优先级：明确公司指标工厂与医院运行包

需要形成正式设计和数据契约：

- 公司指标模板包包含哪些规则、SQL 模板、结构化计算定义、字段概念和测试样例。
- 医院适配包只补哪些字段映射、业务编码和本院口径差异。
- 发布包如何签名、验签、审批、版本化和回退。
- 哪些医院元数据允许回收到公司，哪些绝不出院。
- 公司大模型生成草稿后必须经过哪些自动测试和人工审核。

### 第二优先级：把实施控制台升级为主业务入口

- 固定指标不依赖聊天完成实施。
- 增加指标目录、实施状态、字段映射、测试样例和发布状态总览。
- AI 助手降级为当前页面的解释和诊断辅助。
- 医生只看业务口径、结果和明细；实施人员可展开表字段和 SQL；研发可查看完整 Trace。

### 第三优先级：扩展真实指标适配

- 通过 WxP 与公司数据库核验 `MQSI2025_014`。
- 通过 WxP 与公司数据库核验 `MQSI2025_035`。
- 为每个新增指标建立聚合 SQL、患者/业务明细 SQL、字段血缘、边界样例和记录级对账契约。

### 第四优先级：降低 Ollama 的运行依赖

- 增加“无模型模式”运行开关和前端状态说明。
- 固定业务操作全部走结构化命令。
- 对问答和诊断报告继续保留模板兜底。
- 评估医院是否真的需要部署 4B 模型，还是仅在需要自然语言助手的医院启用。
