## 一、Bug 修复

### 1. `_rule_evidence` 默认 fact_types 缺少 `"rule_identity"`

- **文件**: `app/agent_tools/read_tools.py`
- **问题**: `get_effective_rule` 和 `inspect_indicator_implementation` 返回的 evidence 默认 fact_types 不含 `"rule_identity"`，导致 `has_verified_rule()` 返回 False，`prepare_indicator_sql` 等下游工具被标记为不可用
- **修复**: 在 `_rule_evidence` 默认 fact_types 中增加 `"rule_identity"`

### 2. `prepare_indicator_sql` 被旧 SQL ID 阻塞

- **文件**: `app/agent_tools/sql_tools.py`
- **问题**: 可用性检查 `_state_has_verified_rule` 中包含 `and not has_active_sql(state)`，上一轮遗留的 SQL ID 导致新一轮 prepare 被封死
- **修复**: 移除该条件，允许在规则已验证时随时 prepare 新 SQL

### 3. `search_indicator_rules` 重复调用

- **文件**: `app/agent_tools/read_tools.py`, `app/agent_runtime/contracts.py`
- **问题**: 模型在已确认指标后仍重复搜索，浪费 token 和步骤
- **修复**: 在 `AgentRunState` 中加入 `search_count` 计数器，`search_indicator_rules` 每轮最多调用 1 次（`availability=lambda ctx, state: state.search_count < 1`）

### 4. Response Guard 无限循环

- **文件**: `app/agent_runtime/runner.py`
- **问题**: 中文检查、trial_run 检查、missing_fact_types 检查无纠正次数上限，模型落入死循环直奔 max_steps
- **修复**: 四个 guard 全部加 1 次纠正上限（`chinese_corrections`、`trial_run_corrections`、`fact_corrections`、`evidence_corrections`）

### 5. 异常被静默吞掉

- **文件**: `app/agent_runtime/runner.py`, `app/agent_runtime/service.py`
- **问题**: 多处 `except` 块无日志，异常无法定位
- **修复**: 所有异常路径加 `print(traceback)` 和 `[AGENT-XXX]` 标识日志；工具执行代码、模型调用、流式任务均包裹 try/except

### 6. 模型输出对话格式前缀

- **文件**: `app/agent_runtime/response_guard.py`, `app/agent_runtime/prompts.py`
- **问题**: DeepSeek 模仿历史消息格式输出 "用户:"、"助手:" 前缀
- **修复**: `normalize_agent_answer` 自动剥离前缀；提示词禁止对话格式输出

### 7. 模型选择器在发送时被重置

- **文件**: `web/agent-runtime.js`, `web/index.html`
- **问题**: 每次发送前 `refreshAgentRuntimeMode()` 用服务端默认模型重建下拉框，覆盖用户手动选择
- **修复**: `renderModelSelector` 保留用户已选值；移除 `streamChat` 中的冗余刷新调用

## 五、Planner / Compiler / Controller / Verifier 编排

为提升 4B 本地模型的多步工具调用稳定性，工具调用型 Agent 新增服务端计划编译与受控执行层：

- Planner 只输出业务目标、目标指标、原始时间表达、请求输出、约束和歧义；严格拒绝 `steps`、`proposed_steps`、`tool` 等字段。
- PlanCompiler 根据业务能力依赖生成正式执行计划，不让模型维护具体工具名和调用顺序。
- StateController 根据当前 Evidence 确定下一项业务能力；确定性参数编译器把该能力映射为唯一工具和完整参数。
- TimeRangeResolver 确定性解析本月、上月、今年至今、从某月到现在和明确日期范围。
- PlanVerifier 校验必要事实、分子/分母/指标值、SQL ID、医院、数据源、统计周期与 `context_digest`。
- ReplanPolicy 只允许计划方向错误重规划一次；参数缺失由控制器补步骤或向用户澄清。
- 人工/终止分类包括用户澄清、业务确认、管理员审批、实施支持、系统运维和安全拒绝。

正式 Agent 服务默认启用 `plan_compile_control`；旧聊天、Shadow 分流及其专用意图识别/答案生成提示词已经删除，不再作为回退路径。

## 七、2026-07-17 确定性工具直调

- 新增 `app/agent_planning/dispatch.py`，根据已编译计划、结构化状态和已验证 Evidence 确定性生成工具调用参数。

## 九、2026-07-17 修复上传分析重复调用

- 根因：上传分析工具成功后产生的事实类型是 `upload_analysis`，而计划编译器、状态控制器和证据校验器要求 `file_analysis`，导致成功结果被误判为计划未完成，再次调用相同工具后被重复调用保护终止。
- 上传分析工具现在统一产生规范事实 `file_analysis`，成功后状态控制器直接进入最终回答阶段。
- 新增事实名称兼容层，把历史会话和旧执行记录中的 `upload_analysis` 映射为 `file_analysis`，避免服务升级后旧会话再次触发工具。
- 状态控制器和计划校验器共用同一规范化规则，并新增工具输出、控制器决策和证据校验三类回归测试。
- 自动化验证：Ruff、`compileall` 和完整测试通过，结果为 805 passed、1 skipped。

## 十、2026-07-17 修复最终回答 DSML 泄漏与上传对比流程

- 根因：DeepSeek 在最终回答阶段没有返回结构化 `tool_calls`，而是把 DSML 工具协议和不存在的 `get_indicator_result` 当作普通正文；原 ResponseGuard 只校验证据声明，没有识别工具协议文本，因此前端原样展示。
- ResponseGuard 新增工具协议检测。发现 DSML、`tool_calls`、`invoke` 或类似标记时记录 `TOOL_PROTOCOL_LEAK`、阻止展示并使用集中提示最多纠正一次；重复异常返回安全错误，不执行虚构工具。
- Planner 明确把“上传 Excel 与本院系统指标对比”规划为同时需要 `file_analysis` 和 `trial_result`，后续只补充指标名称或统计时间时不得退化为公式解释。
- Compiler 新增组合证据链：先完成本院指标试运行，再分析上传文件，使 Excel 工具能够直接核对分子、分母和指标率；缺少指标名称或统计周期时由 Validator 在访问业务库前澄清。
- 自动化验证：Ruff、`compileall`、提示词注册测试和完整测试通过，结果为 810 passed、1 skipped。

## 十一、2026-07-17 统一 8B 与 DeepSeek 的统计周期解析

- 根因：8B 在 SQL 追问中输出了带时间文本的 `raw_text`，但没有填写 `start_time/end_time`，该文本又阻止服务端复用上一轮已确认周期；DeepSeek 会填写字段，但把“1 月到 3 月”的结束时间写成 3 月 31 日零点，按左闭右开会漏掉当天数据。
- `AgentPlanningRuntime` 现在优先解析用户本轮时间原文；本轮没有时间时复用结构化会话周期，不再让模型生成的 `raw_text` 覆盖权威状态。
- `TimeRangeResolver` 支持“一月份到三月份”等中文月份范围，并统一生成 1 月 1 日至 4 月 1 日的左闭右开区间。
- Planner 提示词只要求保留时间原文，不再要求模型换算自然语言月份；模型填写的日期不能覆盖服务端确定性结果。
- 自动化验证：Ruff、`compileall`、提示词注册测试和完整测试通过，结果为 814 passed、1 skipped。
